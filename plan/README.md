# Implementation Plan — overview & the quality philosophy

tri-boost is a depth-3 oblivious gradient-boosting library (Rust core + Python bindings) whose models are exactly decomposable into ≤3rd-order purified fANOVA tables. This plan turns the final spec (§00 skeleton + §01–14) into a sequenced, gated build. Its governing idea: **high engineering standards are not a discipline anyone has to remember — they are a property of the order in which work is done.**

Six structural mechanisms enforce that:

- **Gates before features.** The entire gate wall — `clippy -D warnings` with the no-panic set (`unwrap_used`/`expect_used`/`panic`/`unreachable`/`indexing_slicing`/`cast_possible_truncation`), `#![forbid(unsafe_code)]` at the `tri-boost-core` root, `#![deny(missing_docs)]`, `overflow-checks=true` in every workspace-root profile, `cargo fmt --check`, `cargo deny`, the MSRV-1.74 job, doctests, and the grep-gates (`usize`-in-serialized-fields, `HashMap`-in-fit-path, `Box<dyn Error>`-in-public-sigs, `// JUSTIFIED:`-pairing) — lands in **PR #1**, against an empty workspace. The first line of `bin()` physically cannot merge if it panics, is unformatted, or pulls a banned license. Quality is opt-out-impossible, inherited via `[workspace.lints]` where Cargo supports inheritance.
- **Invariant/test-first.** The five lossless checks (Reconstruction / MassConservation / Purity / VarianceSum / ThreeWayEqual, §13.1) and the determinism gate (`{1,2,8}`-thread byte-equality, §13.4) exist as `Result<(), PbError>` functions over hand-built fixtures in **Phase 0** (Gate G0), *before* the engine that produces a `TableBank`. The moment a tree exists they point at it. "Tables == ensemble" and reproducibility are enforced from tree #1, never audited later.
- **Vertical slices.** Every milestone is end-to-end runnable *and* gated — never a pile of untested modules.
- **DoD = gates-green, not code-written.** Every task's Definition of Done names the exact CI gate(s) it must turn green, plus its spec §-ref.
- **Traceability.** Every task cites the spec § it implements; every §13 gate maps to a named CI job (see the matrix).
- **Illegal states unrepresentable.** `PbError`/`Invariant`, the `ExactnessMode` firewall, and construction-time I1 enforcement in `ObliviousTree::new` make a non-exact model or a >3-feature tree a *type error*, not a runtime check.

**Build narrative.** v1 is the **green-spine alpha**: bin → depth-3 oblivious Newton boost on SquaredError → accumulate → purify → `TableBank`, proven *tree-sum == table-sum* and *bit-identical across `{1,2,8}` threads*. Only once the spine is green do we broaden — full loss set, interaction funnel + monotone + `wht8`, categoricals, predictiveness boosters, then the Python/inference/perf layer. v1.5 is the **benchmark release gate**: a dev-only `xtask` harness proves, with explicit hardness, that **every model is `Exact`** (HARD) and **tri-boost beats EBM/GA2M on median held-out deviance** (HARD) on TabArena, while **approach to unconstrained GBDTs** is *reported* and triggers review but does **not** block release (the order-3 cap makes GBDT parity dataset-dependent, not promised) — predictiveness and exactness certified *together, by gates*.

# The test / invariant / determinism / benchmark through-line

The test machinery is not written after the code; it is **the scaffolding the code is poured into**. Three rules bind every milestone area:

1. **The gate is authored in the same PR as the feature it gates** (or before it). Binning ships with its round-trip proptest; the engine ships with the Newton-gain unit test + the determinism harness pointed at the real `fit`; purification ships with the five checks on the real `TableBank`. A feature's PR is *incomplete* until its named gate is green.
2. **The five invariant checks (§13.1) and the determinism gate (§13.4) run over the *entire model corpus* on every PR.** Any milestone that erodes exactness or reproducibility breaks trunk immediately — exactness cannot drift from accuracy because every model the accuracy harness scores must first clear all five checks.
3. **Gates grow, never get rebuilt.** Phase 0's stubs are *filled in*, not replaced: the determinism harness is met first by **fixed-order float folds** on the full-precision green spine (M1), then gains the quantized-integer path (M5-QHIST), then SIMD-dispatch (M5), then the Python FFI mirror (M5); the §13.1 matrix gains one column/arm per loss and per constraint. Purity and VarianceSum are promoted CHECK→**GATE** the moment §08 lands; VarianceSum is gated only under product/uniform `w` (Joint downgrades to a Shapley-sum CHECK).

Cross-cutting through-line deliverables, sequenced into the milestones that first need them: **proptest scaffolding + frozen `ProptestConfig`** (with M2-A); **Criterion benches + baselines** wired at M1 (perf regressions visible from the first engine commit); **cargo-fuzz** targets (`fuzz_deserialize`/`fuzz_binning`) nightly after M5; the **`wht8`↔purification cross-check oracle** (§13.7a, M2-J/M4-12) — a distinct test from the screening accumulator; the **TreeSHAP oracle** (dev-only, M6-5); the **exhaustive-vs-sampled-corners** nightly (§13.12, M2 onward).

# Definition-of-Done template + PR/branch discipline

**The reusable DoD checklist — every task/PR must satisfy all that apply:**

1. **Spec §-ref cited** in PR description (PR template enforces it).
2. **Standing gates green** (always): `cargo fmt --check`; `cargo clippy --all-targets --all-features -- -D warnings` (incl. no-panic set); `cargo test --all-features` + `--no-default-features`; doctests + `deny(missing_docs)`; `cargo deny check`; MSRV-1.74 build; NoPyo3; Wasm smoke; the `usize`/`HashMap`/`Box<dyn Error>`/`// JUSTIFIED:` grep-gates.
3. **The feature's named §13 gate green** — cited explicitly, never "added tests" (e.g. "G3 ThreeWayEqual green over the §13.1 matrix," "§13.4 byte-equal across `{1,2,8}`," "§13.3 domain-error correct-variant"). Authored **in the same PR** as the feature.
4. **The five invariant checks + determinism gate re-run green over the whole model corpus** (automatic — they run on every PR; the bar is they don't regress).
5. **Every fitted model `Exact`** unless the task explicitly exercises the firewall, in which case the `Approximate` path is gated (`ExactnessFirewall` error tested).
6. **Any `unwrap`/`expect`/`#[allow(indexing_slicing)]`** outside the unwrap-allowed set `{tests, benches, xtask}` carries a `// JUSTIFIED:` comment with a bounds proof (grep-gated; the gate excludes those dev-tooling dirs).
7. **Criterion baseline not regressed** beyond threshold ([CHECK], non-blocking but reported) for tasks touching hot paths.

**PR/branch discipline:** small reviewable PRs on feature branches; trunk stays green (all gates) at all times — no PR merges with a red gate. One logical change per PR. The two large risk-bearing tasks (M1.3+M1.4 engine; M1.8 purify+five-checks) are **never** lumped into one PR — split into reviewable sub-PRs each turning a sub-gate green. "Done" is **gates-green, not code-written**: a task whose code compiles but whose named gate is not green is not done.

**Reconciled area disagreements:** (1) The `wht8` cross-check oracle is owned by **M2-J** (authored with §07); **M4-12 references it** as the Faith-Shap cross-check oracle test only.
