# Milestone 5 — Python API, Inference, Performance Hardening & the v1.5 Levers — Implementation Plan

## How this area's structure makes quality automatic

This milestone is the *consumption-plus-levers* layer. Two things live here. First, the **consumption layer**: it never constructs trees or tables, it only marshals into the green core and surfaces it. Second, this is the **pre-benchmark "v1.5 levers" milestone** — it also carries the v1.5 accuracy/scale levers that must exist *before* the M6 whole-product benchmark gate can certify them: Minimal Variance Sampling (`M5-MVS`), sparse-tensor storage for hot triples (`M5-SPARSE`), and quantized-integer histograms (`M5-QHIST`, moved out of the v1 green-spine). M6's release gate certifies these levers; this milestone is where they are built. The same quality machinery covers both halves: every consumption path is a *bit-equal re-encoding* of an already-gated path (delegation plus equality), and every lever re-passes the five invariant gates + the determinism gate with the lever ON.

Quality on the consumption half is enforced by **delegation plus equality**. Three structural facts do the work:

1. **Every new code path is a re-encoding of something already gated, so its DoD is a *bit-equality* test against the canonical path, not a fresh correctness argument.** `ScoringBank`/`PackedTree` (§10.2a), the row-blocked kernel (§11.9), `predict_binned`, and the flat path-B arena are all "load-derived views" — they ship *only* with a proptest asserting byte-identity to `score_trees`/`score_row` at tolerance 0 (§10.7). You physically cannot land the fast path without the equality gate green, so "fast" can never silently mean "different."
2. **The determinism `[GATE]` (§13.4) is the spine that runs through every task.** It already exists from the foundation milestone (the `model_is_bit_reproducible_across_thread_counts` harness, §11.8). This milestone *extends its subject set* — serialized bytes, then SIMD-dispatched scoring, then the Python FFI — but never re-derives it. The byte-equality assertion is the same `bincode::serde::encode_to_vec(.., standard())` comparison everywhere.
3. **The firewall and `PbError`→exception funnel make illegal states unrepresentable across the FFI.** No panic crosses the boundary because the core is no-panic (§1) and the single `map_err` funnel (§12.7) turns every `PbError` variant into a typed Python exception. The binding adds *zero* math, so there is nothing in this layer that can break I1/I2 — they are upheld "purely by delegation" (§12 intro).

Sequencing rule honored throughout: serialization (the determinism gate's wire subject) lands before the packed scoring view (which must prove it leaves that wire form untouched); both land before the Python binding (which delegates to them); SIMD/streaming hardening rides last because it is a *constant-factor view* over already-gated kernels. Tasks reference the foundation milestone's gate machinery (clippy `-D warnings` incl. the no-panic set, fmt, `cargo deny`, MSRV 1.74, doctests, `deny(missing_docs)`, the determinism harness, the proptest invariant suite) rather than re-specifying it. These map to the v1 Phase 6/7 exit gates **G6/G7** (§14.5) plus the v1.5 perf-hardening row.

## v1.5 accuracy/scale levers (built here, certified at the M6 benchmark gate)

These three tasks are the v1.5 levers that must exist before M6 can certify them. They are *not* consumption-layer re-encodings; each is a real engine/storage capability whose DoD is the **five invariant gates + the determinism gate green with the lever ON** (plus the lever's own oracle), exactly like the M4 booster discipline. They preserve the quality-by-structure spine: a lever that erodes exactness or reproducibility breaks trunk immediately.

### M5-MVS — Minimal Variance Sampling (the row sampler that dominates GOSS)
- **§-ref:** §06 (engine/re-seeding), §13.4 (determinism), §13.1 (five checks).
- **Deliverable:** a `Sampling::Mvs` path in the engine — Minimal Variance Sampling, the row sampler that dominates GOSS — selecting the per-round row subset by gradient-magnitude-aware MVS probabilities, **deterministic** via the seeded per-work-unit re-seeding (`splitmix64_mix` with the frozen `Stage`/`block` keying, §1/§06), so the chosen subset and the resulting `Model` are byte-reproducible across `{1,2,8}` threads. The sampler touches only *which rows enter the histogram* per round; it never changes tree shape arithmetic, leaf-refit-from-full-precision, or the decomposition — exactness is preserved by construction. `Sampling::Full` (v1 default) stays the byte-identical inert oracle.
- **Dependencies:** M1 engine (the full-precision histogram/Newton path it samples into).
- **DoD (gates):** the **five invariant gates** (Reconstruction / MassConservation / Purity / VarianceSum / ThreeWayEqual, §13.1) + the **determinism `[GATE]`** (§13.4, byte-equal `Model` across `{1,2,8}`) green with **MVS on**; every fitted model stays `Exact`. **Variance `[GATE]`:** a test that the MVS split-score estimator variance ≤ uniform-subsample variance at equal sampling rate. `Sampling::Full` byte-identical inert-default oracle. clippy/fmt/doctests green. (M4-11 is the §09-side decomposition-safety **gate** pointing at *this* implementation — it verifies MVS preserves exactness/determinism; MVS is a v1.5 item, not v2.)
- **Size:** M

### M5-SPARSE — sparse `Tensor` storage for hot triples + `OverflowPolicy::SparseFallback`
- **§-ref:** §08 (`EffectTable`/`TableBank`/purify/five checks), §13.1.
- **Deliverable:** a **sparse (coordinate/CSR) representation** for `Tensor`, used for hot triples whose dense merged grid would breach the cell budget. `accumulate`/`purify` and the five invariant checks iterate sparse tensors correctly (same canonical routing, same f64 accumulation, same purification math — only the storage differs); serde and `RatingExport` handle the sparse representation (round-trip stable). `OverflowPolicy::SparseFallback` **replaces** the bare `OverflowPolicy::Error` backstop (M1.8) when a table exceeds `max_table_cells`: instead of erroring, the table is stored sparse. The dense path stays the default for small tables.
- **Dependencies:** M1 §08 `TableBank` (the dense accumulate/purify/checks it generalizes) + the M1.8 `TableBudget`/`max_table_cells` budget (the threshold that triggers fallback).
- **DoD (gates):** the **five invariant gates** (§13.1) green over **sparse-stored tensors** (Reconstruction over corners incl. missing, MassConservation, Purity, VarianceSum, ThreeWayEqual to `recon_tol`) + a **sparse round-trip golden** (sparse `Tensor` → serde → reload → byte-stable, scoring bit-identical). `OverflowPolicy::SparseFallback` test: a table over `max_table_cells` is stored sparse (not `Err`), still passes all five checks. clippy/fmt/doctests green. (M6-6's sparse-triple gate points at *this*.)
- **Size:** M

### M5-QHIST — quantized-integer histograms (moved out of M1; v1 uses full precision)
- **§-ref:** §06 (`Hist`/`QuantGradHess`), §11 (perf), §13.4 (determinism), §13.5 (quantization bound).
- **Deliverable:** **quantized-integer histograms** — `quantize -> QuantGradHess` (per-round `g_scale`, **stochastic rounding** re-seeded via `Stage::Quantize`); `QuantizedI32` accumulation (`g_q as i64`, integer-associative ⇒ order-independent); **mandatory full-precision leaf refit** (structure decided on quantized histograms, the 8 leaves refit from full precision). `Config.hist_precision` gains `QuantizedI32`; `FullF64` (the v1 green-spine baseline) is retained as the exact reference. This is **moved out of M1** — the v1 green-spine uses full precision; quantized histograms are the v1.5 speed lever.
- **Dependencies:** M1 engine (the **full-precision baseline** it is cross-checked against and refits leaves from).
- **DoD (gates):** the **determinism `[GATE]`** (§13.4) green via the **integer-associative sums** (order-independent across `{1,2,8}`), **cross-checked equal to the M1 full-precision + fixed-fold path within `recon_tol`** (same chosen splits/scores); **dequantization-error bound (§13.5):** `|dequant(quant(g)) − g| < 0.5 / g_scale` (HALF a quantization step — **not** `0.5 * g_scale`); leaves refit from full precision (asserted bit-identical to the full-precision leaf solve on the chosen structure); subtraction-trick integer test still exact; **~2× speed bench** (Criterion, vs `FullF64`). Five invariant gates stay green on the quantized path; every model `Exact`. clippy/fmt/doctests green.
- **Size:** M

## Sequencing rationale (nothing builds on an unverified foundation)

`T1→T2→T3` establish the **wire contract first** — the determinism gate's subject — so every later "view" can prove it leaves the bytes untouched. `T4` (packed scoring) and `T5` (digitize-once) are gated *bit-equal* against the canonical scorer from `T1`, never landing as unverified modules. `T6` brings up the binding shell with the wheel/exception/`cargo deny` gates green before any Python feature code (gates-before-features at the FFI layer). `T7→T8` deliver the native + sklearn surface, each gated by the Python conformance + cross-FFI determinism mirror of the core gate. `T9` (SIMD/streaming) and `T10` (benches) ride last because they are constant-factor hardening over already-gated kernels — the ISA-byte-equality gate guarantees speed never costs reproducibility, and Criterion baselines make any regression visible from this milestone forward.

The **v1.5 levers** (`M5-MVS`, `M5-SPARSE`, `M5-QHIST`) build on the same already-green spine and inherit the same discipline: each is real engine/storage capability whose DoD is the five invariant gates + the determinism gate green *with the lever ON* (plus its own oracle), so none can land while eroding exactness or reproducibility. `M5-MVS` and `M5-QHIST` build on the M1 full-precision engine (QHIST is cross-checked equal to that baseline within `recon_tol`); `M5-SPARSE` generalizes the M1.8 §08 `TableBank` storage under its `max_table_cells` budget. M6's release gate certifies these levers; this milestone is where they are built (the M6 §09-side gate `M4-11` points at `M5-MVS`, and `M6-6`'s sparse-triple gate points at `M5-SPARSE`).

Foundation/through-line gate machinery (clippy `-D warnings` + no-panic lints, fmt, `cargo deny`, MSRV 1.74, doctests, `deny(missing_docs)`, the `{1,2,8}`-thread determinism harness, the five invariant proptests) is inherited, not re-specified; every task's DoD is *gates-green*, not *code-written*.
