# Milestone 6 — The v1.5 Release Gate (the benchmark milestone)

## How this area's structure makes quality automatic

Milestone 6 is the **whole-product proof**, and its structure enforces that quality is a property of *how the benchmark is run*, not a thing reviewers eyeball afterward. Three structural facts do the work:

1. **The harness is downstream of the gates, never a substitute for them.** Per §1.6 / §11.8 / §13.7, the accuracy harness ships **no library code** — it is a dev-only `xtask` binary that *consumes* a fitted `Model`. It cannot fake exactness: every model it benchmarks must already be `ExactnessMode::Exact` and must pass `assert_exact_decomposition` (§13.1) and `check_feature_budget` (§13.2) as a **precondition of being scored at all**. The **exactness clause** of the release gate — clause **(a)** of the M6-7 milestone, "every fitted model bit-exactly decomposable" — is wired as a hard `[GATE]` *inside the harness loop*, so a predictiveness number is only ever reported for a model that already cleared the five invariant gates. Accuracy and exactness cannot drift apart, because the harness refuses to print a deviance for a non-`Exact` model.

2. **Reproducibility is inherited, not re-litigated.** The benchmark protocol seeds every stage through the frozen `splitmix64`-mix re-seed (§1/§11.3) and runs on the same determinism discipline (`{1,2,8}`-thread byte-equality, §13.4) the spine already enforces. A benchmark run is therefore itself reproducible by construction — a number that moves between runs is a determinism-gate failure, caught upstream, not a "flaky benchmark."

3. **The adversarial fixtures stress the *firewall*, not the *score*.** The table-size benchmark (§14.3, §08.10, §07.5) is structured so that the failure mode is a **typed `PbError::TableBudget`** or the exact sparse-tensor fallback (the **M5-SPARSE** `OverflowPolicy::SparseFallback` path) — never silent memory blowup or a coarsened table that quietly breaks Reconstruction. The benchmark *asserts the firewall held*; quality is the byproduct of the typed wall, not of a memory watcher.

Every task below cites its spec §-ref, names the exact gates it must pass (referencing the foundation areas rather than re-specifying them), and is sequenced so nothing benchmarks on an unverified lever. The **v1.5 release gate cannot be attempted until every accuracy lever is landed and has individually re-passed G3** (§14.3) — those dependencies are enumerated in M6-0.

## Prerequisite gate (the entry condition for this milestone)

**M6-0 — v1.5 lever-completion checkpoint** · §14.1/§14.3, §14.5 map
**Deliverable:** not code — a tracked CI checklist asserting every v1.5 accuracy lever is merged and green on trunk before the harness is allowed to run as a gate. Required-complete, by owning section: **quantized integer histograms** (§06/§11, `M5-QHIST` — `QuantGradHess` + mandatory full-precision leaf refit; v1.5, moved out of M1), **Fisher sorted-ordinal TS + EB shrinkage** (§04), **multi-distinct-categorical-axis trees** (§04/§06), **MVS sampling** (§06, `M5-MVS` — the `Sampling::Mvs` engine path), **sparse Tensor storage for hot triples** (§08, `M5-SPARSE` — `OverflowPolicy::SparseFallback`), **bagged greedy ensemble selection** (§09, default off), and the **§08 `TableBudget`/`SparseFallback`** firewall + the **§07 `table_budget_beta` admission penalty**.
**Dependencies:** all of v1 (G0–G7, GREEN-SPINE ALPHA) + the v1.5 levers above — including the now-real **M5-MVS** (sampler) and **M5-SPARSE** (sparse tensor storage) implementation tasks.
**DoD:** each lever's own v1.5 sub-gate green on trunk (§14.3 per-lever gates); the §13.4 `{1,2,8}`-thread byte-equality gate now green **on the quantized path (M5-QHIST) and with MVS on (M5-MVS)**; `cargo deny`, `clippy -D warnings`, `fmt --check`, MSRV 1.74 (§13.8/§13.9) green. **Until M6-0 is green, M6-3/M6-4/M6-7 are blocked.** Size: **S** (gating/coordination, no new code).

## Sequenced tasks

**M6-1 — `xtask` workspace member + fixture-loader scaffold** · §02.5 (workspace layout), §11.8, §13.7
**Deliverable:** a new `xtask/` workspace member (binary crate, **dev-only, not published, not in the wheel** — `cargo xtask accuracy …`). `tri-boost-core` and `tri-boost-py` gain **zero** dependency on it. A `Fixture` loader for TabArena-style tabular fixtures (csv/parquet via a dev-dep) producing `(BinnedMatrix, y, weight, exposure, FitSpec)` with a declared per-fixture objective (one of `SquaredError`/`Logistic`/`Poisson`/`Gamma`/`Tweedie{rho}`). Fixtures live under `xtask/fixtures/` (small, committed; large ones fetched by a seeded, checksummed script). `unwrap`/`expect` are **permitted** here: `xtask/` is in the foundation's unwrap-allowed set `{tests, benches, xtask}` (dev tooling), so the §13.8 no-panic grep/clippy gate **excludes** `xtask` — no conflict with the grep-gate, which never scans `xtask`.
**Dependencies:** M6-0.
**DoD:** `cargo build -p xtask` green under the workspace lint table; a CI grep gate (§13.4/§13.8) confirms no `xtask` symbol is reachable from `tri-boost-core`/`tri-boost-py` (the "ships no library code" invariant); the no-panic grep/clippy gate scopes itself to the non-`{tests,benches,xtask}` set (so `unwrap`/`expect` in `xtask` do not fail it); `cargo deny` still green with the added dev-deps justified per §1. Size: **M**.

**M6-2 — per-objective metric module: deviance / lift / ordered-Gini** · §05 (`Loss::deviance`, `Metric`), §01.7, §13.7
**Deliverable:** an `xtask` metric module computing, per fixture: (a) **strictly-proper deviance/logloss** by reusing the *core's* `Loss::deviance` (never re-implemented; never RMSE on Poisson/Gamma/Tweedie — §05/§01.7), `?`-propagating its `Result`; (b) **lift** (decile/ordered lift curves on response-space `pred`); (c) **ordered-Gini** (the area-based concentration metric on rank-ordered predictions, the pricing-standard discrimination measure). These are **dev-yardstick metrics only** — they are deliberately *not* added to the shipped `Metric` enum (§05 owns it: `{Rmse,LogLoss,Poisson/Gamma/TweedieDeviance}`), keeping §1.6's "no shipped benchmarking tooling" true.
**Dependencies:** M6-1.
**DoD:** unit tests in `xtask` pin each metric against a hand-built fixture with known deviance/lift/Gini; deviance is asserted to equal `core::Loss::deviance` on shared inputs (single source of truth); a degenerate-input test confirms metrics never produce `NaN`/`Inf` (mirrors §13.5). Size: **M**.

**M6-3 — seeded reproducible benchmark protocol (the run engine)** · §1 determinism, §11.3 re-seed, §13.4, §01.7 (bagged-vs-bagged)
**Deliverable:** the `xtask accuracy` run engine: deterministic seeded train/test splits (the frozen `splitmix64`-mix seeding scheme, §11.3 — *not* an ad-hoc RNG), strong out-of-the-box defaults (§1.7: more, lower-LR trees; deviance early stopping), and the **bagged-vs-bagged** comparison harness (§01.7 — the self-ensemble §09 path vs EBM's ~14× internal bagging). Emits a structured, committed-to-disk results artifact (json) keyed by `(fixture, objective, seed)` with deviance/lift/ordered-Gini per model. **The per-model exactness precondition is wired here:** before any model's metrics are recorded, the engine calls `assert_exact_decomposition` (§13.1) and `check_feature_budget` (§13.2) and **refuses to record a number for a non-`Exact` model** (the M6-7 exactness clause **(a)** as a loop invariant).
**Dependencies:** M6-1, M6-2, M6-0.
**DoD:** a **[GATE]** reproducibility self-test — two independent `xtask accuracy` runs at the same seed produce byte-equal results artifacts (the §13.4 cross-run reproducibility property, applied to the harness); every recorded model has `ExactnessMode::Exact` and passed all five §13.1 checks + §13.2 (asserted in-loop, build-blocking when run in CI nightly mode). Size: **L**.

**M6-4 — rival adapters: EBM/GA2M + unconstrained GBDTs** · §01.7, §11.8, §13.7
**Deliverable:** `xtask`-local adapter shims to fit and score the rivals on the *same* seeded splits and the *same* M6-2 metrics: **EBM/GA2M** (the headline rival, order-2 cap), and **unconstrained XGBoost / LightGBM / CatBoost** (the accuracy ceiling). Rivals run via subprocess/Python-env shells invoked from `xtask` (kept entirely outside the Rust core; no rival dependency enters any shipped crate). Results land in the same M6-3 artifact schema so tri-boost-vs-rival deltas are a pure table read.
**Dependencies:** M6-3.
**DoD:** an adapter smoke test fits each rival on one tiny fixture and produces finite deviance/lift/Gini in the shared schema; a CI grep confirms no rival crate (`xgboost`, `lightgbm`, …) appears in `tri-boost-core`/`tri-boost-py` `Cargo.toml` (the boundary holds); `cargo deny` unaffected (rivals are not Rust deps). Size: **M**.

**M6-5 — TreeSHAP attribution oracle (validates §08 exactness in-harness)** · §13.7, §08 (Faith-Shap/equal-split φ)
**Deliverable:** wire stock **TreeSHAP as a test oracle only** (its sole permitted appearance, §13.7) into the harness: for each benchmarked tri-boost model, compare TreeSHAP attributions against the exact equal-split `φ_i` read from the §08 tables. This is the *external* corroboration of the `ThreeWayEqual` claim, complementary to the §13.7a `wht8`↔purification oracle (which is a separate, core-side `[GATE]` and **not** owned here).
**Dependencies:** M6-3 (needs fitted-model + table-export plumbing).
**DoD:** **[GATE]** TreeSHAP attributions agree with the table-derived `φ_i` to the derived `recon_tol` (§13.1) on the benchmark corpus; a documented assertion that TreeSHAP stays demoted to oracle (never a shipped explainer). Size: **M**.

**M6-6 — table-size adversarial benchmark (border-rich + bagged fixtures)** · §14.3 release-gate, §08.10 (`TableBudget`/`SparseFallback`), §07.5 (`table_budget_beta`)
**Deliverable:** an `xtask accuracy --adversarial` mode with fixtures combining **border-rich axes** (axes deliberately near the 254-bin cap, §03 R-BINS) with **deep bagging** (the §09 ensemble union/merged grid), driving merged-grid 2D/3D supports to worst case. It measures and asserts: **(a)** per-table cell counts respect §08's `max_table_cells` (default 2M) and the total-bank budget — with the §07 high-cardinality admission penalty (`table_budget_beta`) deprioritizing the most expensive triples, and either the §08 **`OverflowPolicy::SparseFallback`** (the **M5-SPARSE** path — exact, the same tensor sparsely stored) or a typed **`PbError::TableBudget { u, cells, budget }`** engaging at the ceiling (never silent truncation); **(b)** purification wall-time stays within a measured envelope on these grids; **(c)** peak memory of the bagged union-grid bank is bounded.
**Dependencies:** M6-3 (run engine), M6-0 (the §08 budget + §07 penalty + the **M5-SPARSE** sparse-fallback must be landed).
**DoD:** **[GATE]** the five §13.1 invariant checks still pass on the **sparse-stored hot triples** (the M5-SPARSE sparse `Tensor` representation; sparse storage is exactness-neutral, §08.10); a budget breach surfaces as `OverflowPolicy::SparseFallback` or `PbError::TableBudget`, never silent memory/wall-clock inflation (asserted); a control assertion that `table_budget_beta = 0.0` reproduces the un-penalized ordering exactly (§07 soft-not-hard property); purification-time and peak-memory envelopes recorded against committed baselines (§13.7 Criterion-style `[CHECK]`). Size: **L**.

**M6-7 — the v1.5 milestone evaluation (the whole-product proof)** · §01.7, §14.3 release gate, §13.1/§13.2 gates
**Deliverable:** the headline evaluation run over the full TabArena-style corpus, bagged-vs-bagged, producing the §01.7 milestone verdict under three clauses with **explicit hardness**: **(a) EXACTNESS [HARD, release-blocking]** — every fitted model in the suite is `ExactnessMode::Exact` and passes all five §13.1 invariant gates + §13.2; **(b) BEAT EBM [HARD, release-blocking]** — tri-boost beats EBM/GA2M on the **MEDIAN held-out per-objective deviance across the suite** (the objective-appropriate deviance/logloss, never RMSE on Poisson/Gamma/Tweedie); **(c) APPROACH GBDTs [REPORTED + triggers review, NOT release-blocking]** — tri-boost lands within a **defined target band of the unconstrained GBDTs** (XGBoost/LightGBM/CatBoost): the per-objective median deviance gap to the best unconstrained GBDT is **≤ 2% relative** (the stated target band); exceeding it is reported and **triggers a review**, but does not block release — the order-3 cap means GBDT parity is dataset-dependent and not promised. The harness emits a per-dataset verdict table and honestly flags the disclosed minority (dense ≥4-way / Higgs-/Year-class targets, §01.7) where the order-3 bias ceiling is expected.
**Dependencies:** M6-3, M6-4, M6-5, M6-6.
**DoD:** **clause (a) is a hard `[GATE]`** — the run fails if *any* fitted model is non-`Exact` or fails any of the five §13.1 checks or §13.2 (the firewall and the gate are one wall, §13.1). **Clause (b) is a hard `[GATE]`** — the run fails if tri-boost does **not** beat EBM/GA2M on the corpus-**median** held-out per-objective deviance (release-blocking). **Clause (c) is a reported `[CHECK]`** — the median-deviance gap to the best unconstrained GBDT is recorded per objective against the **≤ 2% relative** target band; out-of-band results are flagged and trigger a review but are **not** release-blocking (dataset-dependent yardstick, per §01.10). Size: **L**.

**M6-8 — release checklist & publication gate (v1.5 RELEASE GATE)** · §02.7 (CI matrix, wheels, publishing), §13.9/§13.10, §14.3
**Deliverable:** the executable release checklist binding the milestone verdict to publication. (1) **All gates green on the full CI matrix** (§13.10): `{stable, MSRV 1.74} × {linux-x86_64, linux-aarch64, macos-aarch64, windows-x64}` running fmt/clippy `-D warnings`/test/doctests/determinism (§13.4)/the five invariant gates (§13.1–2)/`cargo deny`, both `--all-features` and `--no-default-features`. (2) **Wheels build across the matrix** via `maturin-action` (abi3-py310, manylinux2014/manylinux_2_28 + musllinux, macOS x86_64/aarch64, Windows x64) + sdist; the §13.10 clean-env wheel smoke test (fit/predict/serialize) passes **before** any PyPI publish. (3) **`tri-boost-core` is crates.io-publishable** (`cargo publish -p tri-boost-core --dry-run` green; `cargo-semver-checks` `[CHECK]` against the §2 canonical types, §13.9). (4) **Docs complete** — `#![deny(missing_docs)]` is a compile gate (§13.8) so public-item doc coverage is 100% by construction; `--doc` tests every `///` example; the milestone-verdict summary (M6-7) and the §10 "score in SQL"/`RatingExport` recipe are present.
**Dependencies:** M6-7 (its two hard gates — clause (a) exactness and clause (b) beat-EBM-median — green are the precondition to publish; clause (c) approach-GBDT is reported, not a publish blocker).
**DoD — the v1.5 EXIT CRITERION:** the §01.7 milestone met on TabArena under the M6-7 hardness semantics — **(a) [HARD]** every fitted model passes the five invariant gates (§13.1) and is `ExactnessMode::Exact` (§13.2); **(b) [HARD]** tri-boost beats EBM/GA2M on the corpus-median held-out per-objective deviance; **(c) [REPORTED]** the approach-the-unconstrained-GBDTs gap is recorded against the ≤ 2% relative target band and reviewed if out-of-band (NOT release-blocking); full CI matrix green (§13.10); wheels build + clean-env smoke pass on every `(os,arch)`; `tri-boost-core` publishable to crates.io via OIDC Trusted Publishing (§02.7); the table-size adversarial gate (M6-6) green. **This is the whole-product proof — exactness (a) and beat-EBM predictiveness (b) certified together, by gates, not by audit; GBDT parity (c) is honestly reported, not promised.** Size: **M**.

## Sequencing rationale

M6-0 is the firewall against benchmarking an unverified foundation: the milestone is *defined* as the levers-complete state (§14.3), so no harness task may run as a gate until every v1.5 lever (categoricals/§04, quantized hist + MVS/§06, ensemble/§09, table budget/§08, admission penalty/§07) is merged and has re-passed G3 individually. M6-1→M6-2→M6-3 build the harness bottom-up (scaffold → metrics → seeded run engine), with the per-model exactness precondition wired into the run engine itself (M6-3) so it is structurally impossible to report accuracy for a non-exact model. M6-4/M6-5 add rivals and the external oracle on top of a run engine that is already reproducible and exactness-gated. M6-6 (adversarial table-size) and M6-7 (headline evaluation) are independent stresses on the same engine, and M6-8 binds both to publication: the release gate cannot pass unless **clause (a) exactness is green AND clause (b) beat-EBM-median is green** (M6-7's two hard gates) **and** the budget firewall held under stress (M6-6) **and** the full CI matrix, wheels, and crates.io publishability are green — the v1.5 RELEASE GATE as a single, gated exit. Clause **(c)** (approach the unconstrained GBDTs, within the ≤ 2% relative target band) is **reported and triggers review when out-of-band, but is not release-blocking** — the order-3 cap makes GBDT parity dataset-dependent and not promised.

**Key finding for the caller:** ordered-Gini and lift are *not* in the shipped `Metric` enum (§05 owns it as `{Rmse, LogLoss, Poisson/Gamma/TweedieDeviance{rho}}`); they live only in the dev-only `xtask` metric module (M6-2), which keeps §1.6's "ships no benchmarking tooling" invariant intact. The repo contains the **spec** (`/home/ralph/suite/tri-boost/spec/`) **and this implementation plan** (`/home/ralph/suite/tri-boost/plan/`); **M6 builds on M1–M5** — by the time this milestone runs, the workspace already holds `crates/` (the green core + Python binding) and the v1.5 levers (incl. **M5-MVS**, **M5-SPARSE**, **M5-QHIST**). M6-1 creates the `xtask/` workspace member per the §02.5 layout (`/home/ralph/suite/tri-boost/spec/02-architecture.md` lines 25–55) on top of that already-built core.
