# Milestone 6 ‚Äî The v1.5 Release Gate (the benchmark milestone)

## How this area's structure makes quality automatic

Milestone 6 is the **whole-product proof**, and its structure enforces that quality is a property of *how the benchmark is run*, not a thing reviewers eyeball afterward. Three structural facts do the work:

1. **The harness is downstream of the gates, never a substitute for them.** Per ¬ß1.6 / ¬ß11.8 / ¬ß13.7, the accuracy harness ships **no library code** ‚Äî it is a dev-only `xtask` binary that *consumes* a fitted `Model`. It cannot fake exactness: every model it benchmarks must already be `ExactnessMode::Exact` and must pass `assert_exact_decomposition` (¬ß13.1) and `check_feature_budget` (¬ß13.2) as a **precondition of being scored at all**. Clause (c) of the ¬ß1.7 milestone ‚Äî "every fitted model bit-exactly decomposable" ‚Äî is wired as a hard `[GATE]` *inside the harness loop*, so a predictiveness number is only ever reported for a model that already cleared the five invariant gates. Accuracy and exactness cannot drift apart, because the harness refuses to print a deviance for a non-`Exact` model.

2. **Reproducibility is inherited, not re-litigated.** The benchmark protocol seeds every stage through the frozen `splitmix64`-mix re-seed (¬ß1/¬ß11.3) and runs on the same determinism discipline (`{1,2,8}`-thread byte-equality, ¬ß13.4) the spine already enforces. A benchmark run is therefore itself reproducible by construction ‚Äî a number that moves between runs is a determinism-gate failure, caught upstream, not a "flaky benchmark."

3. **The adversarial fixtures stress the *firewall*, not the *score*.** The table-size benchmark (¬ß14.3, ¬ß08.10, ¬ß07.5) is structured so that the failure mode is a **typed `PbError::TableBudget`** or the exact sparse-tensor fallback ‚Äî never silent memory blowup or a coarsened table that quietly breaks Reconstruction. The benchmark *asserts the firewall held*; quality is the byproduct of the typed wall, not of a memory watcher.

Every task below cites its spec ¬ß-ref, names the exact gates it must pass (referencing the foundation areas rather than re-specifying them), and is sequenced so nothing benchmarks on an unverified lever. The **v1.5 release gate cannot be attempted until every accuracy lever is landed and has individually re-passed G3** (¬ß14.3) ‚Äî those dependencies are enumerated in M6-0.

## Sequenced tasks

**M6-1 ‚Äî `xtask` workspace member + fixture-loader scaffold** ¬∑ ¬ß02.5 (workspace layout), ¬ß11.8, ¬ß13.7
**Deliverable:** a new `xtask/` workspace member (binary crate, **dev-only, not published, not in the wheel** ‚Äî `cargo xtask accuracy ‚Ä¶`). `tri-boost-core` and `tri-boost-py` gain **zero** dependency on it. A `Fixture` loader for TabArena-style tabular fixtures (csv/parquet via a dev-dep) producing `(BinnedMatrix, y, weight, exposure, FitSpec)` with a declared per-fixture objective (one of `SquaredError`/`Logistic`/`Poisson`/`Gamma`/`Tweedie{rho}`). Fixtures live under `xtask/fixtures/` (small, committed; large ones fetched by a seeded, checksummed script). `unwrap`/`expect` are permitted here (¬ß13.8 exempts `xtask`).
**Dependencies:** M6-0.
**DoD:** `cargo build -p xtask` green under the workspace lint table; a CI grep gate (¬ß13.4/¬ß13.8) confirms no `xtask` symbol is reachable from `tri-boost-core`/`tri-boost-py` (the "ships no library code" invariant); `cargo deny` still green with the added dev-deps justified per ¬ß1. Size: **M**.

**M6-2 ‚Äî per-objective metric module: deviance / lift / ordered-Gini** ¬∑ ¬ß05 (`Loss::deviance`, `Metric`), ¬ß01.7, ¬ß13.7
**Deliverable:** an `xtask` metric module computing, per fixture: (a) **strictly-proper deviance/logloss** by reusing the *core's* `Loss::deviance` (never re-implemented; never RMSE on Poisson/Gamma/Tweedie ‚Äî ¬ß05/¬ß01.7), `?`-propagating its `Result`; (b) **lift** (decile/ordered lift curves on response-space `pred`); (c) **ordered-Gini** (the area-based concentration metric on rank-ordered predictions, the pricing-standard discrimination measure). These are **dev-yardstick metrics only** ‚Äî they are deliberately *not* added to the shipped `Metric` enum (¬ß05 owns it: `{Rmse,LogLoss,Poisson/Gamma/TweedieDeviance}`), keeping ¬ß1.6's "no shipped benchmarking tooling" true.
**Dependencies:** M6-1.
**DoD:** unit tests in `xtask` pin each metric against a hand-built fixture with known deviance/lift/Gini; deviance is asserted to equal `core::Loss::deviance` on shared inputs (single source of truth); a degenerate-input test confirms metrics never produce `NaN`/`Inf` (mirrors ¬ß13.5). Size: **M**.

**M6-3 ‚Äî seeded reproducible benchmark protocol (the run engine)** ¬∑ ¬ß1 determinism, ¬ß11.3 re-seed, ¬ß13.4, ¬ß01.7 (bagged-vs-bagged)
**Deliverable:** the `xtask accuracy` run engine: deterministic seeded train/test splits (the frozen `splitmix64`-mix seeding scheme, ¬ß11.3 ‚Äî *not* an ad-hoc RNG), strong out-of-the-box defaults (¬ß1.7: more, lower-LR trees; deviance early stopping), and the **bagged-vs-bagged** comparison harness (¬ß01.7 ‚Äî the self-ensemble ¬ß09 path vs EBM's ~14√ó internal bagging). Emits a structured, committed-to-disk results artifact (json) keyed by `(fixture, objective, seed)` with deviance/lift/ordered-Gini per model. **The per-model exactness precondition is wired here:** before any model's metrics are recorded, the engine calls `assert_exact_decomposition` (¬ß13.1) and `check_feature_budget` (¬ß13.2) and **refuses to record a number for a non-`Exact` model** (clause (c) as a loop invariant).
**Dependencies:** M6-1, M6-2, M6-0.
**DoD:** a **[GATE]** reproducibility self-test ‚Äî two independent `xtask accuracy` runs at the same seed produce byte-equal results artifacts (the ¬ß13.4 cross-run reproducibility property, applied to the harness); every recorded model has `ExactnessMode::Exact` and passed all five ¬ß13.1 checks + ¬ß13.2 (asserted in-loop, build-blocking when run in CI nightly mode). Size: **L**.

**M6-4 ‚Äî rival adapters: EBM/GA2M + unconstrained GBDTs** ¬∑ ¬ß01.7, ¬ß11.8, ¬ß13.7
**Deliverable:** `xtask`-local adapter shims to fit and score the rivals on the *same* seeded splits and the *same* M6-2 metrics: **EBM/GA2M** (the headline rival, order-2 cap), and **unconstrained XGBoost / LightGBM / CatBoost** (the accuracy ceiling). Rivals run via subprocess/Python-env shells invoked from `xtask` (kept entirely outside the Rust core; no rival dependency enters any shipped crate). Results land in the same M6-3 artifact schema so tri-boost-vs-rival deltas are a pure table read.
**Dependencies:** M6-3.
**DoD:** an adapter smoke test fits each rival on one tiny fixture and produces finite deviance/lift/Gini in the shared schema; a CI grep confirms no rival crate (`xgboost`, `lightgbm`, ‚Ä¶) appears in `tri-boost-core`/`tri-boost-py` `Cargo.toml` (the boundary holds); `cargo deny` unaffected (rivals are not Rust deps). Size: **M**.

**M6-5 ‚Äî TreeSHAP attribution oracle (validates ¬ß08 exactness in-harness)** ¬∑ ¬ß13.7, ¬ß08 (Faith-Shap/equal-split œÜ)
**Deliverable:** wire stock **TreeSHAP as a test oracle only** (its sole permitted appearance, ¬ß13.7) into the harness: for each benchmarked tri-boost model, compare TreeSHAP attributions against the exact equal-split `œÜ_i` read from the ¬ß08 tables. This is the *external* corroboration of the `ThreeWayEqual` claim, complementary to the ¬ß13.7a `wht8`‚Üîpurification oracle (which is a separate, core-side `[GATE]` and **not** owned here).
**Dependencies:** M6-3 (needs fitted-model + table-export plumbing).
**DoD:** **[GATE]** TreeSHAP attributions agree with the table-derived `œÜ_i` to the derived `recon_tol` (¬ß13.1) on the benchmark corpus; a documented assertion that TreeSHAP stays demoted to oracle (never a shipped explainer). Size: **M**.

**M6-6 ‚Äî table-size adversarial benchmark (border-rich + bagged fixtures)** ¬∑ ¬ß14.3 release-gate, ¬ß08.10 (`TableBudget`/`SparseFallback`), ¬ß07.5 (`table_budget_beta`)
**Deliverable:** an `xtask accuracy --adversarial` mode with fixtures combining **border-rich axes** (axes deliberately near the 254-bin cap, ¬ß03 R-BINS) with **deep bagging** (the ¬ß09 ensemble union/merged grid), driving merged-grid 2D/3D supports to worst case. It measures and asserts: **(a)** per-table cell counts respect ¬ß08's `max_table_cells` (default 2M) and the total-bank budget ‚Äî with the ¬ß07 high-cardinality admission penalty (`table_budget_beta`) deprioritizing the most expensive triples, and either the ¬ß08 **`SparseFallback`** (exact, same tensor sparsely stored) or a typed **`PbError::TableBudget { u, cells, budget }`** engaging at the ceiling (never silent truncation); **(b)** purification wall-time stays within a measured envelope on these grids; **(c)** peak memory of the bagged union-grid bank is bounded.
**Dependencies:** M6-3 (run engine), M6-0 (the ¬ß08 budget + ¬ß07 penalty must be landed).
**DoD:** **[GATE]** the five ¬ß13.1 invariant checks still pass on the **sparse-stored hot triples** (sparse storage is exactness-neutral, ¬ß08.10); a budget breach surfaces as `PbError::TableBudget`, never silent memory/wall-clock inflation (asserted); a control assertion that `table_budget_beta = 0.0` reproduces the un-penalized ordering exactly (¬ß07 soft-not-hard property); purification-time and peak-memory envelopes recorded against committed baselines (¬ß13.7 Criterion-style `[CHECK]`). **M6-7 ‚Äî the v1.5 milestone evaluation (the whole-product proof)** ¬∑ ¬ß01.7, ¬ß14.3 release gate, ¬ß13.1/¬ß13.2 gates

## Sequencing rationale

M6-0 is the firewall against benchmarking an unverified foundation: the milestone is *defined* as the levers-complete state (¬ß14.3), so no harness task may run as a gate until every v1.5 lever (categoricals/¬ß04, quantized hist + MVS/¬ß06, distillation + ensemble/¬ß09, table budget/¬ß08, admission penalty/¬ß07) is merged and has re-passed G3 individually. M6-1‚ÜíM6-2‚ÜíM6-3 build the harness bottom-up (scaffold ‚Üí metrics ‚Üí seeded run engine), with the per-model exactness precondition wired into the run engine itself (M6-3) so it is structurally impossible to report accuracy for a non-exact model. M6-4/M6-5 add rivals and the external oracle on top of a run engine that is already reproducible and exactness-gated. M6-6 (adversarial table-size) and M6-7 (headline evaluation) are independent stresses on the same engine, and M6-8 binds both to publication: the release gate cannot pass unless clause (c) is green (M6-7) **and** the budget firewall held under stress (M6-6) **and** the full CI matrix, wheels, and crates.io publishability are green ‚Äî the v1.5 RELEASE GATE as a single, gated exit.

**Key finding for the caller:** ordered-Gini and lift are *not* in the shipped `Metric` enum (¬ß05 owns it as `{Rmse, LogLoss, Poisson/Gamma/TweedieDeviance{rho}}`); they live only in the dev-only `xtask` metric module (M6-2), which keeps ¬ß1.6's "ships no benchmarking tooling" invariant intact.
