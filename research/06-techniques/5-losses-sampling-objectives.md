## Losses, sampling & objectives

This section covers the parts of a GBM that sit **on top of**, or **beside**, the tree builder: the loss/objective (its gradient and Hessian), how leaf values are computed, how rows are sampled, how the learning rate is scheduled, when training stops, and per-feature penalties that bias which features enter. The recurring theme is that **almost all of these are orthogonal to tree shape** — they touch the per-row `(g, h)` that feed the histograms, the leaf-value formula, or the boosting loop, none of which know or care that the tree is a depth-3 oblivious lookup table. That orthogonality is exactly why they are easy wins: a new loss is a new `(g, h)` pair; a new sampler is a row mask + reweight; a schedule is a per-round scalar. The handful of genuinely tricky calls are (a) **zero-Hessian losses** (L1/quantile/Huber) that need a per-leaf value *refresh* rather than a Newton step, (b) **multi-output uncertainty heads** that produce two table sets per output, and (c) **`max_delta_step`**, which is two different mechanisms wearing one name.

### The v1 loss/objective core: Newton split gain + boost-from-average + offsets

The non-negotiable v1 spine is the **second-order (Newton) split gain** `Σ_leaves G²/(H+λ)` with leaf weights `w = −G/(H+λ)`, on top of which every objective below is just a new `(g, h)` pair. Two clarifications matter because the source library docs muddle them:

- **`score_function` vs `leaf_estimation_method` are different knobs.** CatBoost's [`score_function`](https://catboost.ai/en/docs/concepts/algorithm-score-functions) only *ranks splits* (its CPU default is the angular `Cosine`); the `−G/(H+λ)` leaf values come from [`leaf_estimation_method=Newton`](https://catboost.ai/docs/en/references/training-parameters/common). tri-boost should default to the **NewtonL2 split gain** (`ΣG²/(H+λ)`, the XGBoost/LightGBM canonical) for parity, which CatBoost happens to restrict to GPU as an engineering choice, not a mathematical one. The `Cosine` angular score is a genuine research A/B, not a v1 commitment.
- **`boost_from_average` IS the fANOVA constant term.** Initializing `F₀ = argmin_c Σ loss(yᵢ, c)` is not just a convergence trick — for a log-link Poisson model with exposure it is `F₀ = log(Σyᵢ / Σeᵢ)`, the deployable base rate, and the `f₀` of the decomposition ([XGB intercept tutorial](https://xgboost.readthedocs.io/en/stable/tutorials/intercept.html)). **Two correctness mandates:** (a) compute it in **score/link space, exposure-weighted**, never `log(mean(y))`; (b) for losses with no closed form (Gamma, Tweedie, Huber) find it by one Newton step `F₀ = −G/H` over the scalar using the existing gradient machinery, with guards for degenerate inputs (all-zero exposure, single-class). Note that with purification, the *reported* `f₀` after the cascade differs from this seed (purification moves mass down into the intercept) — the seed anchors, it does not equal the final intercept; the sum stays bit-for-bit exact.

`base_margin`/`init_score` is the same machinery used two ways: as the **exposure offset** (`offset = log e`, added to `F` on the link scale every iteration — supplying it *disables* the auto mean-init, matching XGBoost/LightGBM) and as **boost-on-top-of-a-GLM** (CANN-style): set `offset = η_GLM(x)` so the trees model only the GLM residual. The latter is a strong pricing deployment story but carries an explainability caveat — the exact tables then explain the *residual*, and the full audit trail is `η_GLM(x) + tables(x)`; only safe if the GLM backbone is itself ≤3-order and additive (guard against a filed GLM carrying a 4-way interaction).

### `max_delta_step`: one name, two different mechanisms — pick deliberately

This is the most error-prone item in scope. The pricing objectives (Poisson/Gamma/Tweedie, log link) have an exponential gradient that can blow up the Newton step when the Hessian is tiny. Two libraries "fix" it differently and **they do not produce the same leaf values**:

- **XGBoost — hard clamp.** After computing `w = −G/(H+λ)`, clip `|w| ≤ max_delta_step` (verified `Sign(ret)·max_delta_step` in `param.h`). Objective-agnostic, but **non-smooth**, and the clamp interacts with monotonicity bound-propagation — a clamped leaf can land outside a propagated bound, so define ordering (enforce monotone bounds *after* clamping) and test it.
- **LightGBM — Hessian inflation.** For Poisson it instead inflates the Hessian to `h = μ·exp(max_delta_step)` (default 0.7), shrinking `|−G/(H+λ)|` smoothly. This is **objective-coupled** and *changes which splits are chosen* (gains differ), so it under-steps the true Newton optimum deliberately — the exported relativities reflect the regularized step.

**Recommendation:** make the **Hessian-inflation route the default for log-link count objectives** (it is the production default in both ecosystems, stays Newton-consistent and smooth, composes more cleanly with monotone bounds), and additionally expose the **hard clamp** for the imbalanced-logistic use case (XGBoost recommends `1–10`). Keep the global default `0` (off) so squared-error/AUC benchmarks match XGBoost/LightGBM; auto-set `0.7` only for `count:poisson`. fANOVA exactness is unaffected either way — both only touch leaf values. The bit-for-bit reconstruction test must compare against whatever weight is *actually emitted*.

### Zero-Hessian losses (L1 / quantile / Huber): the leaf-refresh path

The pinball loss `L_α = max(α(y−F), (α−1)(y−F))` and absolute error have **gradient = piecewise constant `±α` and true Hessian = 0**. You cannot Newton-step. The standard fix (LightGBM/CatBoost/XGBoost `reg:quantileerror`/`reg:absoluteerror`) is: grow the tree structure with `h=1` (so split gain degrades to a first-order `G²` form), then **refresh each leaf** to the weighted empirical α-quantile of in-leaf residuals (median for L1). This requires machinery the Newton path doesn't have:

- an inverse **row→leaf map** plus a **weighted-quantile routine** (a distinct code path, but cheap), and
- it is **not histogram-summable** — you cannot accumulate a median from `(G, H)` bins, so you need per-row residuals partitioned into the 8 leaves (an extra pass).

**Critical fANOVA correction:** the refreshed leaf is still **constant per cell**, so this **preserves exactness** — it is *not* the linear-leaf "changes_table_form" case (a recurring mislabel; median ≠ linear). Each fitted α is a **separate** ≤3-order table set, so multi-quantile intervals ship *N* table stacks, not one — a real explainability/UX decision (a median rating table *and* a 95th-percentile rating table), not a free feature. Watch:

- **Quantile crossing:** independently-fit α's can cross (the 0.9 model predicting below the 0.5 model); XGBoost documents this. Post-hoc isotonic-in-α rearrangement (Chernozhukov) if monotone quantiles are required for VaR/capital.
- **Monotone constraints don't compose cleanly:** bound-propagation assumes Newton leaf weights, but the post-hoc refresh overwrites them — monotone-in-feature guarantees are not automatic and need a constrained refresh or isotonic post-fit.
- **L1 is a pricing hazard:** it targets the conditional *median*, but pure premium needs the *mean*. On right-skewed severity the median sits below the mean, so L1 systematically **under-prices** — document it as robust point/interval estimation, **not** pure-premium estimation; use Gamma/Tweedie for that.

Implement L1, quantile, and Huber's tail together as one "weighted-quantile leaf refresh + optional per-leaf override" feature, since the same hook serves all three.

**The smoother alternatives that dodge the refresh entirely:** [**Expectile**](https://catboost.ai/docs/en/concepts/loss-functions-regression) loss `|α − 1{y<F}|·(y−F)²` has a **strictly positive, piecewise-constant Hessian** (`(1−α)` or `α`, verified in XGBoost's `ExpectileRegression::GetGradient`), so it drops straight into the Newton split-gain + leaf-weight path — no refresh, no special code. It is the *cleanest* asymmetric objective for a second-order booster and serves cost-asymmetric pricing (under-pricing costs more than over-pricing) directly. The **arctan/Huberized smooth pinball** similarly restores a non-vanishing Hessian for quantile estimation at the cost of a smoothing parameter `s` that biases the quantile by `O(s)`. Note that XGBoost's *non-crossing* multi-expectile guarantee comes from a **reparameterized-gaps vector-leaf representation** (first margin = smallest expectile, rest = positive gaps), *not* from the loss itself — two independent single-α oblivious models can still cross, so tri-boost would not inherit non-crossing for free.

### Robust losses: the link-mismatch trap

Huber, Pseudo-Huber, Log-Cosh, Fair, Lq, Cauchy are all drop-in `(g, h)` swaps, oblivious-native, table-preserving. The substantive caveat across the whole family is **link mismatch**: they operate on raw residuals `(y − F)` under an **identity link** (Gaussian regression). tri-boost's headline objectives are **log-link** (Poisson/Gamma/Tweedie), where "residual" and the transition threshold `δ` live on the multiplicative scale and the constant-Hessian assumption no longer matches the curvature. So plain Huber/Log-Cosh compete with squared-error use-cases, **not** with the Gamma/Tweedie severity path. For robust *severity on the response scale* the apples-to-apples options are quantile/pinball, a Huberised-Gamma-deviance, or simply Gamma/Tweedie themselves. Secondary notes:

- **[Pseudo-Huber](https://xgboost.readthedocs.io/en/stable/parameter.html)** `δ²(√(1+(r/δ)²)−1)` is smooth with `h = (1+(r/δ)²)^(−3/2) > 0`, but `h→0` for large `|r|` ([XGBoost #9378](https://github.com/dmlc/xgboost/issues/2602) documents non-convergence — "fitted value stuck at a large number"), so it still wants a Hessian floor / `max_delta_step`. Better-conditioned near 0, *worse*-conditioned far out.
- **Log-Cosh** is parameter-free but `h = sech²(r)` decays exponentially (underflows f32 around `|r|≈20`), starving split points for heavy-tail rows — needs an ε-Hessian floor — and has a hidden fixed scale (transition at `r=1` in target units), so it needs target standardization, undermining the "no hyperparameter" pitch.
- **Cauchy/Lorentzian** is redescending and **non-convex** (Hessian goes negative for large `|r|`); only usable with the per-leaf `Σh ≥ ε` safeguard (the same trick **Robust-GBDT** relies on for noisy-label robustness). Research-only, for genuinely contaminated data.

### Sampling: ship Bernoulli in v1, MVS in the perf pass, skip GOSS

Row sampling only changes which rows feed the per-level `(G, H)` histograms (with reweighting) — **fully orthogonal** to the oblivious split search and the tables. The hierarchy is clear:

- **v1: plain Bernoulli `subsample`** — simplest regularizer+speedup. But the "~1/subsample" speed claim is the *histogram-build* bound; real gain is sub-linear (~1.3–1.8× at 0.5–0.66) because full-data gradient/prediction passes remain, and with `sampling_frequency=PerTreeLevel` the histogram-subtraction trick gets messier (different row sets per level) — prefer **PerTree** for the speed win. **Pricing risk:** uniform subsampling is *exposure/Hessian-blind* — on low-frequency Poisson/Tweedie data it drops rare positive-claim rows and inflates tail-leaf variance (the exact cells actuaries audit). Keep rates conservative (0.66–0.8).
- **Perf pass: MVS, not GOSS.** [Minimal Variance Sampling](https://arxiv.org/abs/1910.13204) samples row `i` with `p_i = min(1, √(gᵢ² + λ·hᵢ²)/μ)` (μ a quickselect threshold for the budget) and reweights by `1/p_i`. **Correct formula note:** `mvs_reg` weights the **Hessian** term `λ·hᵢ²` — a common record error writes `√(gᵢ² + mvs_reg)`, which builds the wrong sampler and loses the second-order advantage MVS exists to give (paper Thm 2). MVS **provably dominates GOSS** on split-score estimator variance (GOSS uses one fixed probability for all non-top rows; MVS adapts per row), so **implement MVS and skip GOSS** — GOSS's only subtle part, the `(1−a)/b` reweight, is superseded by MVS's `1/p`. Under the strongly varying pricing Hessians, the `(g² + λh²)` importance is *more* impactful but also more sensitive — validate that exposure-offset/Tweedie `hᵢ` magnitudes don't destabilize `μ`.
- **Both are non-deterministic** and must be **seeded with a stable per-level RNG stream** — the lossless-equivalence invariant test and the regulatory "re-derive the exact table" requirement both depend on it.
- Bayesian bootstrap (`bagging_temperature`, CatBoost's *default*) is a pure per-row reweight `w = (−log u)^t` — variance reduction with **no** speedup, so it is redundant with subsample for v1; nice-to-have for matching CatBoost's untuned robustness. CatBoost itself notes MVS "may not be the best choice for regularization" (sampled rows are similar across close iterations), so don't expect MVS to double as a regularizer — pair it with explicit `λ`/`min_data`.

### `min_*` floors and `min_gain_to_split`: the oblivious level-veto

These are standard but acquire a sharper edge under obliviousness. Because **one shared split is applied to all 2^k cells at a level**, a `min_sum_hessian_in_leaf` / `min_data_in_leaf` test must hold across **all 8 leaves simultaneously** — a single under-supported cell **vetoes the split for the entire level**, strictly more aggressive than the per-node test in leaf-wise libraries, and able to starve depth-3 growth faster. This is a feature for pricing (every one of the 8 rating cells gets minimum credibility) but defaults must account for the 8-way partition, not be copied from leaf-wise libraries.

Two refinements worth designing in:
- **Make `min_data_in_leaf` exact.** LightGBM's is *approximate* (Hessian-derived, occasionally violated); the histogram already carries an exact per-bin count, so tri-boost can gate on the true count — free, and more defensible to a regulator.
- **Credibility ≠ Hessian mass.** With exposure, the Poisson Hessian is `h = e·exp(F) = e·μ` (exposure × *current predicted rate*), which drifts every iteration and is **not** a raw exposure floor. For a stable actuarial credibility floor, prefer `min_data` on raw count, or add a **weight-sum (exposure) floor** `Σeᵢ` — a small tri-boost-specific addition.

`min_gain_to_split`/`gamma` gates whether a whole level's best shared split is worth adding; **a level failing the threshold simply isn't added, yielding a `<depth-3` (lower-order, `<3`-feature) tree** — a legitimate, fANOVA-friendly outcome that must be a *tested code path*, not an error. Note this is a **design extension**: CatBoost's `SymmetricTree` grows to *fixed* depth and does not do gamma-style early level-stopping, so tri-boost must define the semantics (recommend: terminate the tree at the failing level). Default `0` (off) for parity. The speed pitch is marginal here — the per-level scan over an ≤8-leaf histogram tensor is tiny; the value is regularization and lower-order trees, not speed.

### Leaf-value refinement: multi-step Newton, backtracking, path smoothing

These act only on the 8 leaf values of a *fixed* oblivious structure, so all preserve exactness:

- **`leaf_estimation_iterations > 1`** takes multiple Newton steps per tree, re-deriving `(g, h)` at the updated leaf values each step — a real accuracy lever for **non-quadratic pricing losses** (Gamma/Tweedie). If recomputed on leaf-aggregated sums it's cheap; for arbitrary-loss per-leaf optima it may need per-row re-evaluation per iteration, so verify the cost path before defaulting `>1` (CatBoost defaults to 1 for most regression). Keep **Newton**, not CatBoost's first-order `Gradient` default which is strictly worse for non-quadratic losses; `Exact` is only a closed form for **L1-type** losses, so it buys nothing for v1 Poisson/Gamma/Tweedie and is deferred-and-loss-contingent.
- **`leaf_estimation_backtracking`** (Armijo / AnyImprovement) is a line search guaranteeing the step doesn't worsen loss — robustness hardening for ill-conditioned losses (Tweedie with small variance power), only relevant when iterations `>1`.
- **[`path_smooth`](https://lightgbm.readthedocs.io/en/latest/Parameters.html)** shrinks each leaf toward its parent by sample count: `w·n/(n+ps) + w_parent·ps/(n+ps)`. This **smooths thin/low-exposure cells** — directly the credibility/smoothness need in insurance rating — and stays constant-per-cell. "native" is slightly overstated: it needs explicit construction of the recursive root-to-leaf parent path (and, for LightGBM parity, threading the smoothed prior into the per-level gain), not just a leaf post-step; the prior is the ancestor split-node value, **not** a designed actuarial credibility complement, so treat it as a *foundation* for a credibility-shrinkage feature. Verify the reconstructed-table invariant still holds with it on, and that shrinkage doesn't nudge a leaf across a monotonicity bound (enforce constraints *after* smoothing). Implement from LightGBM's *code* form, not the [old docs](https://github.com/microsoft/LightGBM/issues/5139).

### Early stopping and learning-rate schedules: loop-only, but mind the export

Both govern the boosting loop, not tree shape, so they are structurally inert — yet they have a real interaction with the **lossless-equivalence invariant**. With early stopping the model object typically retains *all* trained trees but predicts on the `best_iteration` prefix; **the fANOVA table export must accumulate exactly that same prefix.** Accumulating all trained trees while predicting on a prefix would make reconstructed tables ≠ served model and silently fail the bit-for-bit test. The XGBoost native-vs-sklearn `predict` inconsistency (sklearn auto-uses `best_iteration`, native `Booster.predict` uses the full model unless given `iteration_range`) is a documented footgun — pick **one** semantics (recommend: predict defaults to the trimmed prefix) and align export to it.

Two more points:
- Ship **plain patience early stopping in v1** (identical across XGBoost `early_stopping_rounds` / LightGBM callback / CatBoost `od_type=Iter`) — it is *not* a CatBoost differentiator. CatBoost's adaptive **IncToDec p-value test** is at best a v1.5/v2 refinement (opaque `od_pval` threshold, CatBoost-only).
- **Early-stop on the offset-aware deviance**, not raw loss — for Poisson/Gamma/Tweedie with exposure the validation metric must be the per-objective deviance (XGBoost's `tweedie-nloglik` takes the response-scale `μ` while the objective takes raw `F` — same math, different input space, easy to get wrong), or the stopping point is biased. Pin the deployed iteration count for reproducibility, and run any auto-calibration/balance post-step *after* truncation.
- **Learning-rate schedules** (`LearningRateScheduler`, BetaBoosting-style shaped `η_t`) are trivial — scale each tree's 8 leaves by `η_t` before accumulation — and fully exact, but the payoff over a tuned constant `η` is **uncertain/mixed**; a v2 convenience, not a differentiator. **DART** (tree dropout) is a genuine anti-over-specialization regularizer that keeps the ensemble a sum of oblivious tables, but it bakes **non-unit per-tree scale weights** into the leaves — fold those into the exported tables (a closed-form rescale) so the rating table stays clean, and note it destabilizes early stopping (no running-sum buffer).

### Per-feature penalties and CEGB: parsimony levers (with an ordering rule)

`feature_weights`, `first_feature_use_penalties`, `per_object_feature_penalties` ([CatBoost](https://catboost.ai/docs/en/references/training-parameters/common)) and CEGB (`cegb_tradeoff`, `cegb_penalty_feature_coupled/lazy/split`, [LightGBM](https://lightgbm.readthedocs.io/en/latest/Advanced-Topics.html)) all **adjust the per-level split score** before the argmax — directly compatible with the summed-gain search, no table-form change. They serve two real pricing goals: **feature-set parsimony** (`first_feature_use_penalties` → fewer, cleaner tables for an actuary) and **cost-aware modeling** (prefer cheap/already-acquired rating factors; CEGB charges per-forest, per-row-lazy, or per-split feature-acquisition costs). Caveats:

- These are **soft priors, not hard caps.** With hundreds of trees a useful feature usually pays back its one-time penalty and still enters; for a strict feature-count cap you need an explicit constraint, not a penalty.
- **Define order of operations** with monotone/interaction constraints (handled elsewhere): **reject constraint-violating candidates first, then apply weights/penalties to survivors** — so a penalty can never resurrect a constrained-out split.
- Apply the multiplicative weight to the **gain magnitude** (positive), not a signed/near-zero gain, or down-weighting can invert ordering near zero. Match CatBoost's non-negative semantics (`feature_weights = 0` ⇒ effective drop).
- Under the oblivious structure CEGB's "first use of a feature" / `num_data_in_leaf` semantics degenerate (a level-1 feature is shared by all 8 leaves), so the coupled penalty becomes a flat per-feature tax — fine, but no longer "CEGB" in a meaningful sense. The [LightGBM `left_count > 0` crash](https://github.com/microsoft/LightGBM/issues/5825) under `cegb_penalty_feature_coupled` is a reason to design natively rather than port. Research-priority; most pricing datasets fetch all factors anyway.

### Stochastic structural regularizers: `random_strength`, `extra_trees`, `gradient_based`

`random_strength` adds a decaying Gaussian to each candidate split score (tie-breaking, anti-greedy-overfit), and `extra_trees` evaluates only **one random threshold per candidate feature**. Both preserve exactness (still a single shared `(feature, threshold)` per level, still constant leaves) but must be **re-derived for the per-level summed-gain loop** (one random draw / jitter applied to the level-wide gain over all 2^k leaves, *not* per-node), making `extra_trees` "adaptable" rather than native. Two practical warnings: (a) the `random_strength` σ must be **scaled to the per-level gain spread**, not an absolute constant — Poisson/Gamma/Tweedie gains live on wildly different Hessian/exposure-dependent scales, so a fixed σ is either a no-op or destabilizing; (b) a single random border shared across a whole level is a **coarser** perturbation than LightGBM's per-node draw, so empirical variance reduction differs and must be benchmarked. Both need seeded, level-indexed RNG for reproducible/auditable rating splits, should be **off by default**, and stack with bagging/`colsample` as additional randomness sources (tune jointly). `gradient_based` subsampling ([XGBoost](https://xgboost.readthedocs.io/en/stable/parameter.html), `p_i ∝ √(g² + λh²)`) is the same importance-sampling idea as MVS and a v2 speed knob once the core is fast.

### Out of scope, recorded for completeness

- **Ranking objectives** (LambdaMART, `rank:ndcg`/`rank:map`/`lambdarank`, position-bias correction, `sampling_unit=Group`) are mechanically compatible — pairwise λ-gradients are just a `(g, h)` source feeding the oblivious Newton build, tables stay exact — but there is **no query-group ranking task in rating-table production**. High plumbing cost (group ids, ΔNDCG machinery, position debiasing) for zero pricing payoff. **Skip** unless a concrete ranking use case appears.
- **`survival:aft` / `survival:cox`** are standard second-order objectives (gradients from the chosen noise distribution / partial likelihood) that plug into the oblivious build with no structural change and stay ≤3-order. Genuinely relevant to insurance (policy lapse/retention, time-to-claim, large-loss development) but niche vs. v1 frequency/severity — **research/objective-library** tier, gated on interval-censored label metadata.
- **`scale_pos_weight` / class weights / focal / Robust-Focal** address imbalanced *classification* (rare-peril, fraud flags). All are per-row or per-class reweights / `(g, h)` swaps and oblivious-clean, but pricing is dominated by Poisson/Gamma/Tweedie regression and the same effect is reachable via the general `sample_weight` path tri-boost needs anyway. Focal's closed-form Hessian can go **negative** (non-convex in the margin) and the loss is **improper** (classification-calibrated but not a proper scoring rule — needs post-hoc recalibration before any probability/risk use), so it requires a Hessian floor and a recalibration step; **v2 at most**. `scale_pos_weight` is the simpler default; both interact with `min_sum_hessian_in_leaf` (thresholds gate on *weighted* H) and break total balance, so pair with a predict-then-recalibrate step.
- **Poisson bootstrap** is a GPU-only implementation convenience; CPU-first ⇒ **skip** (Bernoulli/MVS cover the same need on CPU).
