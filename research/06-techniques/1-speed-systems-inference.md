## Speed, systems & inference

This section covers the techniques that make pattern-boost *fast* and *deployable*: training-speed acceleration (gradient-based row sampling, quantized/integer-gradient histograms, column subsampling, GPU), inference (branch-free vectorized oblivious lookup, table-sum scoring, compilation via Treelite/lleaves/ONNX, leaf/threshold quantization), data-pipeline & memory layout (pre-quantized matrices, midpoint/subsampled binning, deterministic parallel reduction), distributed/out-of-core training, incremental/refit/warm-start, and prediction-time controls. The single load-bearing observation for this whole section: **a depth-3 oblivious ensemble is unusually friendly to every "regular kernel" systems technique** (one shared split per level ⇒ one tiny histogram tensor, one branch-free 3-bit gather), AND it *already eliminates* the irregular-traversal cost that most third-party inference compilers exist to remove — so several famous inference systems (QuickScorer, HummingBird, Treelite, lleaves) are *moot* rather than incompatible.

### The core inference win: branch-free SIMD oblivious lookup + table-sum scoring

The structural payoff of the depth-3 oblivious constraint *is* inference speed, and pattern-boost should treat this as a v1 deliverable since predict-speed parity is an explicit goal.

**Mechanism.** For a single oblivious tree, all rows take the *same three* splits in the *same order*, so there is zero branch divergence. Per level `j` you compute the comparison bit for a whole batch of rows with one packed SIMD compare, then `idx[i] |= bit_j[i] << j`, building a 3-bit leaf index in `{0..7}`. The final step adds `leaf[idx[i]]` — and because the leaf table is only **8 floats (32 bytes, L1-resident)**, you do *not* need a true gather: on AVX2 an in-register permute/blend over the 8-float LUT is faster than a hardware gather ([CatBoost evaluator](https://catboost.ai/news/best-in-class-inference-and-a-ton-of-speedups)). This is the key micro-kernel detail: *3 packed compares → OR-shift bits → in-register permute, not gather.*

**Why the verdict (`native`, `preserves`) holds.** Prediction stays a *sum of per-tree LUT lookups*, additive across trees, so the table decomposition is untouched. Two adoption guardrails:

- **fp32 leaves are the default and are mandatory for the exactness invariant.** The reported AVX2 fp32 path (~20–40% over scalar) is lossless; the larger ~50–70% AVX-512 figure *requires float16 leaves*, which the source itself ([arXiv:2211.00391](https://arxiv.org/abs/2211.00391), not 2405.11062 — fix the citation) calls a new speed/quality trade-off. fp16 leaves **break** the bit-for-bit "reconstructed tables == ensemble" invariant and perturb the audited rating relativities. Expose fp16 only as a separately-flagged *approximate* scoring mode.
- Never let a "fused multi-tree" SIMD variant couple >3 features or break per-tree additivity — that would silently violate the order-3 cap at the inference layer.

**LUT-sum inference** is the pattern-boost-specific superpower that goes *beyond* tree evaluation. Because the trained model is *exactly* `f0 + Σ T1[i][b_i] + Σ T2[ij][b_i,b_j] + Σ T3[ijk][b_i,b_j,b_k]` on the shared quantization grid, you can deploy and predict from the **tables**, not the ensemble — EBM ships exactly this way ([EBM docs](https://interpret.ml/docs/ebm.html)). Bin each feature once (binary search on borders), then sum a handful of dense small-array lookups; the lookup count per row is the number of *nonempty* tables (≈ distinct realized feature triples, usually ≪ C(p,3)), **independent of tree count**. This collapses an M-tree ensemble into O(#tables) loads and is strictly cheaper than M oblivious-tree evals once M is large. It also makes external compilers (Treelite/lleaves/ONNX) largely unnecessary, since they target *traversal* that pattern-boost has eliminated.

> **Hard correctness footgun (must be a tested invariant).** Table inference is lossless *only* if it uses the **full** set of realized feature supports — every ≤3-feature triple that ever appeared as a tree. The explanation pipeline *prunes* tables (FAST/variance/heredity selection); predicting from the curated display subset would NOT reproduce the ensemble. Keep two views: the **complete tensor set for inference**, the **pruned set for human display**. Equality is to floating-point tolerance (summation order differs), so reuse the per-cell reconstruction + variance-sum checks as CI invariants. Additivity lives on the **raw/link scale**: for log-link pricing, multiplicative relativities require *sum-then-exp* (per-table `exp` factors multiply) — a presentation detail, not a break.

**Prediction caching / memoization** (skip) is the trivially-true part folded into the baseline: bin-once amortizes across all terms, and a 1D term is a single 256-entry array load. Tuple memoization of `(region, vehicle-group, age-band)` cells is a deployment *export* convenience (precompute a flat cross-tab for fully-categorical models) but **must not** become a mutable score-time cache — that breaks the deterministic SIMD-across-rows batch scorer and adds an invalidation/reproducibility hazard for no measured win.

### Why most third-party inference compilers are moot (not incompatible — redundant)

This is the trickiest *negative* set of calls, and it's worth stating plainly so future work doesn't re-propose them.

- **QuickScorer / RapidScorer** ([SIGIR'15](http://ecmlpkdd2017.ijs.si/papers/paperID718.pdf), [KDD'18](https://ai.stanford.edu/~wzou/kdd_rapidscorer.pdf)) win by *avoiding traversal of deep irregular trees* — their cost scales with the number of false internal nodes. A depth-3 oblivious tree has only 3 internal splits, already evaluated branch-free in 3 compares; there is *nothing to save*. Crucially, do **not** confuse pattern-boost's bit-packed leaf index with QuickScorer's leaf-elimination bitvector — they are *opposite* computations. Bit-packing computes the leaf index in O(depth) with no masks; QuickScorer accumulates a per-tree elimination bitvector via masks and scans for the survivor, which only pays off at large depth/leaf count.
- **Treelite/TL2cgen** ([MLSys 2018](https://mlsys.org/Conferences/doc/2018/196.pdf)) compiles branchy traversal into nested if-else; a hand-written branch-free LUT evaluator beats it. (Note the project split: modern Treelite 4.x is now a serialization/exchange format; the C-codegen moved to TL2cgen, and CatBoost/oblivious ingestion is still an open issue, [#377](https://github.com/dmlc/treelite/issues/377).)
- **lleaves** ([siboehm](https://github.com/siboehm/lleaves)) LLVM-compiles per-node code (10–30× over *interpreted* LightGBM; ~9.7s→0.4s on MTPL2, a real pricing dataset). Same logic: it accelerates traversal that pattern-boost has removed. Its MTPL2 number is a **speed bar to beat**, not a dependency. The genuinely transferable *idea* — JIT-compiling the model to native code — should target pattern-boost's *own* LUT-sum kernel via Cranelift/LLVM if intrinsics fall short, as a separate research item, not via lleaves.
- **HummingBird / tensor-compiled forests** ([OSDI'20](https://www.usenix.org/conference/osdi20/presentation/nakandala), [survey arXiv:2302.04430](https://arxiv.org/pdf/2302.04430)) reformulate forests as GEMM to get a *regular* kernel for *irregular* trees. Oblivious trees are *already* regular (fixed 3-compare + 8-LUT), and LUT-sum is leaner still. Keep only the survey's design *lesson*: **vectorize across rows, batch, watch small-batch latency** — and apply it to pattern-boost's own batched leaf-index gather (a v1.5 inference-kernel note), not by adopting HummingBird.
- **ONNX export** (v2) is the one with real value, but *not* for speed: it's a portability/MLOps artifact for enterprises standardized on ONNX serving. Caveats: the ONNX `TreeEnsemble` op is a *generic binary tree*, so the export is **structurally lossy** (loses obliviousness + the order-3 cap), the legacy op is float32-only (`~3e-5` threshold discordance — a real concern under the bit-for-bit invariant; force `*_as_tensor` double / op-5 where possible), and for an oblivious predictor ONNX is more likely speed-*neutral-to-negative*. Position it as a secondary interop target, never as a substitute for table/rating-table export.

### Training speed: row & column sampling

Row sampling only changes *which* rows feed the per-level gradient/hessian histograms, so it is uniformly `native` and exactness-preserving — the only subtlety is reweighting and determinism.

**Bernoulli subsample (v1)** is the simplest lever and the right first ship. Two honest caveats: the "~1/subsample" speedup is the *histogram-build bound*; real gains are sublinear (~1.3–1.8× at 0.5–0.66) because full-data gradient/prediction passes remain, and with `PerTreeLevel` the histogram-subtraction trick gets messier (different row sets per level) — **prefer `PerTree` for the speed win**. For Poisson/Gamma/Tweedie with exposure offsets and low claim rates, aggressive uniform subsampling drops rare positive-claim rows and inflates tail-leaf variance (the exact cells actuaries audit) — keep rates conservative (0.66–0.8) or move to gradient-aware sampling. Subsampling must *multiply*, not replace, existing sample/exposure weights, and needs a seeded, stream-ordered RNG to keep the lossless-equivalence test stable.

**MVS (v2) is the sampler to actually build, not GOSS.** MVS samples row `i` with probability `p_i = min(1, sqrt(g_i² + λ·h_i²)/μ)` — note the **hessian** term `λ·h_i²` (the finder's `mvs_reg` formula dropped it; this is the second-order signal MVS exists to provide, [paper Thm 2](https://arxiv.org/abs/1910.13204)) — with `μ` found by an O(N) quickselect so `Σp_i = N·subsample`, and reweights survivors by `1/p_i`. It provably minimizes the variance of the L2 split-score estimate and **strictly dominates GOSS** (GOSS uses one fixed probability for all non-top rows; MVS adapts per row). Hence **GOSS = skip, MVS = v2**: implement MVS instead. The `1/p_i` reweighting flows into `h_i`, so it feeds `min_sum_hessian_in_leaf`/`min_child_weight` and the Newton denominators — handle exactly like sample weights, and verify it composes with exposure offsets (it does mathematically, but it's an easy bug). Under the strongly-varying Tweedie/Poisson hessian, MVS importance is *more* impactful but `μ` is *more* sensitive — validate stability. XGBoost's `sampling_method=gradient_based` (`sqrt(g²+λh²)`-proportional, subsample as low as ~0.1) is the same family and a reasonable v2 alternative on the hist path.

**Column subsampling** (`colsample_bytree`/`bylevel`, `rsm`, `feature_fraction`) maps to *per-level* feature subsampling for oblivious trees (3 independent draws/tree; `bynode` collapses into `bylevel` since there is one node per level). It is cheap regularization, but the *speed* claim is overstated: the per-level scan is already row-count-independent after binning, so you only save the scan unless you also skip *building* histograms for unselected features. The real depth-3 risk is **starvation**: with only 3 split opportunities and a hard 3-feature/tree budget, aggressive subsampling can mask the few informative features across all three levels — far more damaging than in a deep leaf-wise tree. Use a higher effective-fraction floor than LightGBM defaults, default to 1.0, and never mask out a feature an interaction-constraint group requires. Note also that XGBoost forbids combining `colsample_by*` with `feature_weights` and that `bynode` is unsupported by the exact method — respect both if exposed. `sampling_frequency`/`sampling_unit` and `bagging_temperature`/Bayesian bootstrap are weaker, regularization-only knobs (v2); the GPU-only Poisson bootstrap is skip (CPU-first; MVS/Bernoulli cover the need).

### Quantized histograms & deterministic reductions (the speed + reproducibility two-fer)

**Quantized / integer-gradient training (`use_quantized_grad`, [Shi-Ke NeurIPS 2022](https://arxiv.org/abs/2207.09682)) is the highest-value training-speed adoption (v1.5).** Per iteration, map `g_i` (and `h_i`) onto a low-bit integer grid scaled by `max|g|` using **stochastic rounding** (`E[round(x)] = x`, so the quantized sum is an *unbiased* estimator — this is what preserves accuracy where deterministic rounding fails), accumulate `int16/int32` histograms, compute split gains in integer arithmetic, then **refit leaf weights from the original full-precision gradients** to remove quantization bias. Reported up to ~2× end-to-end / ~3.8× histogram speedup at ≤2–3 bits with negligible accuracy loss.

The reason this is *especially* attractive for pattern-boost: **integer sums are associative, hence order-independent, hence bit-reproducible** — which directly serves the "tables == ensemble to floating point" tested invariant and the regulated-pricing reproducibility requirement. Adoption corrections to the finder's record:

- It is **not CPU-only** — LightGBM supports it on `cpu` *and* `cuda` (only plain OpenCL `gpu` is excluded).
- **Leaf refit is opt-in and objective-dependent** (`quant_train_renew_leaf` default `false`), and the paper found refit *crucial for regression/ranking* — i.e. exactly pattern-boost's Poisson/Gamma/Tweedie path. **Turn refit ON** for the primary objectives.
- `num_grad_quant_bins` is a real accuracy knob, not free; very low bins (2) can degrade complex models, and there was a degenerate-split crash in LightGBM's low-bit path ([#5982](https://github.com/microsoft/LightGBM)→fixed in #6092) — integer-accumulator width and degenerate-split handling need care in-house.
- Reproducibility requires seeding the stochastic-rounding RNG; once seeded, the integer sums are bit-reproducible (a genuine audit plus).

**Deterministic mode (v1)** is the companion discipline. The *honest* guarantee is narrow: LightGBM's `deterministic` + `force_row_wise/col_wise` gives identical results for *same data + params + build* (including across thread counts) but **not** across compilers/platforms/versions — FMA contraction, libm differences, SIMD reassociation, and `-ffast-math` all break it. So **do not claim "bit-reproducible for regulators"; claim "reproducible for a fixed build + environment,"** which is what governance actually requires. The right pattern-boost route is *not* to replicate `force_row_wise` (whose 2× Dataset-memory cost is a float-histogram artifact) but to combine: (a) **per-thread private, cache-line-padded histograms** with a fixed-order `fold(||zeros, accumulate).reduce(||zeros, merge)` (no `Mutex`, no false sharing — the ≤8-leaf tensor makes per-thread buffers tiny/L2-resident), and (b) **integer-quantized histograms** for order-independent (associative) sums. Float (non-quantized) g/h reductions are *not* bit-reproducible run-to-run because rayon merges in work-steal order; so "deterministic" holds for the integer path or an explicit fixed-`par_chunks` merge. Cross-build reproducibility additionally needs disabled FMA-contraction/fast-math and pinned binning/RNG paths. Pin row order, bin-edge computation on subsampled quantiles, level-argmax tie-breaking, and all sampling seeds.

### Binning & memory layout (build the quantized matrix as the core data structure)

pattern-boost should make the **pre-binned columnar `u8` matrix the canonical training structure** — this is the CPU-relevant half of XGBoost's `QuantileDMatrix` (skip the dependency, adopt the layout): compute the quantile sketch once, store only the binned representation, and **reuse the train-derived bin edges for validation/test** (the one real footgun — `max_bin` must match and you must `ref=train` or "information is lost in quantisation"). Persist the bin edges in the model and reuse them at predict/score time; **never re-quantize valid/test against their own quantiles.** (`QuantileDMatrix`'s GPU ELLPACK ~4× saving is irrelevant CPU-first; in a from-scratch Rust core storing only the binned matrix there is no redundant copy to remove anyway.)

Within binning, four cheap correctness/quality wins, all `preserves`-exactness (they set the *grid*, never the order):

- **`max_bin` (v1)** is the granularity of every numeric rating-table axis; the *union* of borders sets each 1D table's row count. Trained models are *not* bit-identical across `max_bin` — that's a feature for pricing (coarser bins = more readable, more stable relativities) but must be a documented, per-feature-tunable choice. With monotone constraints, raise `max_bin` to avoid over-shallow trees. Reserve one bin index for missing (cap usable data bins at 255 for `uint8`).
- **Quantile borders on a ~200k subsample (`bin_construct_sample_cnt`, v1)** removes a full-data sort from setup at negligible accuracy cost. Two musts: **seed the RNG and persist the chosen borders** (they are the table axis — a pricing/audit requirement), and **sample with replacement proportional to weight then compute unweighted quantiles** (frequency semantics, [sklearn binning.py](https://github.com/scikit-learn/scikit-learn/blob/main/sklearn/ensemble/_hist_gradient_boosting/binning.py)) so exposure offsets give correct borders.
- **Midpoint borders for low-cardinality numerics (v1)** — when distinct-count < `max_bins`, place borders at midpoints of consecutive distinct values: exact splits, one bin per distinct value, and **table rows align 1:1 with real feature values** for ordinal factors (age-band, NCD years, vehicle group). Cheap and human-aligned; ensure the reserved missing bin still fits.
- **Border-selection objective (v1.5/research)** — default to `Median`/equal-count quantiles for predictable, even-row tables; expose `GreedyLogSum` (CatBoost-parity) and the **Hessian-weighted quantile sketch** ([XGBoost weighted sketch](https://xgboost.readthedocs.io/en/stable/tutorials/model.html)). The latter is `research`: it is objective-dependent (must be re-sketched against the chosen loss, breaking *bin-once* unless frozen on the init-round hessian), and for log-link objectives whose init hessian is near-constant in F its gain is small — a *static* freeze-once variant is the only one worth prototyping.

`save_binary`/`two_round` I/O caching is pure pipeline QoL (v2). **External-memory training (v2/research)** is a clean structural fit — depth-3 oblivious needs only per-level histograms, which accumulate batch-by-batch in *one pass*, so column-block paging slots straight onto the binned `u8` store — but most pricing data fits in RAM, so defer; the **cache-aware prefetch buffer** for the row→bin gather is the one piece worth grabbing earlier as an in-core CPU win. **GPU histogram build + ELLPACK and `device=cuda`** (research) reinforce the "defer GPU, keep the layout GPU-friendly" decision: the pre-binned `u8`/ELLPACK layout *is* the fast-CPU layout, and the depth-3 histogram tensor (≤8 leaves × features × bins) is unusually small and GPU-resident. Keep this behind a `Backend` trait; key takeaway from the GPU-specific record is that **NewtonL2 is GPU-only in CatBoost purely as an engineering choice — pattern-boost should implement the Newton/Hessian gain on CPU regardless**, with a deterministic summation order. **Distributed histogram parallelism** (feature/data/voting + Reduce-Scatter) is skip: it composes fine (the tiny oblivious tensor makes voting-parallel almost trivial) but multi-node is out of scope for single-machine CPU pricing.

### Inference-time controls, refit, and incremental learning

**Prediction controls (v1).** `Exponent` `prediction_type` is the log-link inverse that turns leaf sums into multiplicative relativities — core to the rating story. Classify it precisely: `ntree_start/ntree_end`/`staged_predict`/`iteration_range` are the **exactness-preserving** knobs (a prefix = sum of fewer ≤3-feature tables; store trees contiguous-by-round so slicing is O(1)); `Exponent`/`Probability`/`LogProbability`/`Class` transform the *output* and are deliberately *non-additive* — never claim the additive table decomposition holds on the post-link output. **Early-stopping interaction (invariant):** if `predict` uses only the `best_iteration` prefix, the fANOVA table accumulation **must** sum exactly that same prefix — accumulating all trained trees while predicting on a prefix violates the lossless-equivalence test. `pred_leaf` (the 3-bit leaf code) is cheap and useful as a *diagnostic*, but its headline GBM+LR leaf-embedding stacking use-case is an **anti-feature**: building a stacked predictor makes the composite ≠ the exact table decomposition and destroys the "tables ARE the deployable rating table" thesis — document `pred_leaf` as diagnostic-only and discourage stacking.

**`refit` / continued training (v1.5/v2)** is a strong operational fit: insurers refresh rates on new experience but want the agreed table *structure* frozen for governance. `refit` recomputes only the 8 leaf values per fixed oblivious structure, so order-3 exactness is preserved — but **two operational requirements**: (1) after refit you must **re-purify** (purified tables are *not* a fixed linear function of raw leaves under the chosen reference measure, so blending previously-purified tables and assuming identity is wrong; losslessness still holds but must be re-derived); (2) `init_model`/warm-start requires the **persisted bin edges to be reloaded identically**, or re-binning a new Dataset mismatches the grid. Use the model's actual objective (Poisson/Gamma/Tweedie with exposure offsets) for the leaf recompute and re-project monotone constraints, or a "recalibrated" model can silently violate monotonicity. Pair refit with structure-staleness/drift monitoring — it recalibrates magnitudes but never moves a threshold or swaps a feature.

**Incremental / online add–delete GBDT (research).** Genuinely interesting for pricing (periodic refresh, GDPR right-to-erasure) but flagged carefully: the published in-place method ([arXiv:2502.01634](https://arxiv.org/abs/2502.01634)) assumes greedy per-node trees with *stale-derivative + split-sampling* approximation — it produces an *unaudited approximation* of the retrain (a regulatory concern even though the table form survives if trees stay constant-leaf), and its per-node subtree-retrain has **no oblivious analog**, hence `incompatible` as published. The fANOVA verdict is genuinely *unclear*: if updates only re-fit leaf values of fixed structures, exactness holds; if they re-pick splits, the tables change. For a regulated pricing product the better target is **exact** GBDT unlearning (DeltaBoost, SIGMOD'23) rather than the approximate online method — route erasure/refresh requirements there.

### Multioutput speed (SketchBoost) — relevant only if multioutput heads land

**SketchBoost** ([NeurIPS 2022](https://arxiv.org/abs/2211.12858)) accelerates *multioutput* GBDT by sketching the `n×d` gradient matrix to `k≪d` columns (Top-Outputs / Random Sampling / Random Projections), so per-split scoring scales with `k` not `d` (up to >40× on 355-class data). The scope correction is decisive: **the O(d) it cuts is in the number of OUTPUTS, not rows/features** — for single-output regression or binary classification (`d=1`) there is nothing to sketch and **zero benefit**, so it does not touch pattern-boost's main pricing path. It becomes relevant only if multioutput heads (multiclass, multi-peril frequency/severity, vector Tweedie/quantile) ever land, in which case it preserves exactness *per output* (each output gets its own ≤3rd-order table set sharing one oblivious structure). `research`, gated on a multioutput roadmap; if pursued, Random Projections is the best-accuracy variant to prototype, benchmarked specifically on the weaker depth-3 learner.

### The one technique that breaks exactness: post-training leaf/threshold quantization

**Leaf-value + threshold quantization** ([arXiv:2510.26557](https://arxiv.org/pdf/2510.26557), TDK InTreeger) maps cleanly onto the LUT export (1D/2D/3D tables *are* leaf-value arrays on binned axes, and threshold quantization is automatic from binning), giving 4–8× smaller models and integer-only edge inference. But **quantizing leaf values breaks the "tables == ensemble to floating point" invariant** — so it must be an explicit, opt-in, separately-named *lossy export mode* emitting a distinct compact artifact, **never the default audited model**, and never inside the train/decompose/verify path (hence `skip` for the canonical product). Two *adjacent, genuinely-lossless* ideas from the same papers deserve separate consideration: (a) **fixed-point/integer reinterpretation of leaf values with integer threshold comparison** (InTreeger/FlInt style) gives integer-only *exact* inference — preserves bit-for-bit reconstruction, possible SIMD/edge benefit, zero fANOVA cost; and (b) **ToaD's training-time feature/threshold-reuse regularizers** penalize introducing new split features/thresholds, which shrinks the merged-threshold grid `|Ω_i|` and the number of distinct ≤3-feature supports — making the fANOVA tables *genuinely smaller and more readable with no precision loss*, squarely on-mission, and its pointer-less bit-packed global-LUT layout maps naturally onto the 8-value oblivious leaf tables. Those two are worth their own v1.5/v2 records; only value-quantization-as-default is the exactness break to avoid.
