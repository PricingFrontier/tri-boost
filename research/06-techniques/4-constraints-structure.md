## Constraints & structure (rating-table fit)

This section covers the levers that *shape* the model's functional form — what features may interact, in which direction effects may move, how smooth they must be, and which splits are forced or forbidden — and how each keeps pattern-boost's exact `constant + 1D + 2D + 3D` table decomposition clean, sparse, and compliant. These are the techniques that turn a generic order-≤3 oblivious GBM into a *deployable rating table*. Crucially, almost all of them act on the **split-candidate set** or on **leaf values**, never on tree shape, so they compose natively with the per-level summed-gain oblivious search and (with two flagged exceptions) preserve fANOVA exactness.

### Monotone constraints — the v1 anchor, but re-derived for the symmetric layer

Monotonicity is a hard pricing/regulatory requirement (premium non-decreasing in risk exposure, etc.), so this is core v1. The mechanism in all three reference libraries is identical in spirit: during split finding a candidate split on a constrained feature is **rejected** unless the induced child-value ordering respects the sign, and **bound propagation** carries `[lower, upper]` constraints down the tree so descendant leaves cannot violate the order ([XGBoost monotonic tutorial](https://xgboost.readthedocs.io/en/stable/tutorials/monotonic.html)).

**Why the standard mechanism does not port verbatim.** XGBoost/LightGBM bound propagation is built for *asymmetric per-node* trees: each split affects one node, and bounds recurse along independent root-to-leaf paths via a parent/sibling-midpoint rule. In an oblivious tree there is **one shared split per level applied to all `2^k` nodes simultaneously**, so the feasibility check and the bound clamp must hold *jointly across the whole symmetric layer*. The correct adaptation (the CatBoost approach) is to treat the 8 leaf values as a single vector and impose **joint pairwise constraints** on every pair of leaves that differ only in the constrained-feature bit — i.e. "constrained leaf-value optimization on the shared-split table," not literal midpoint propagation. Spell out the midpoint rule for the `2^level → 2^(level+1)` expansion: each parent-leaf's bound splits to its two children with the midpoint as the shared boundary, and the level's summed gain is `-∞` if *any* cousin pair would violate.

**Two failure modes sharper than in asymmetric trees.** (1) **Empty-level failure:** because the constraint must hold across all `2^k` cells at once, a monotone-violating shared split sets the *entire level's* summed gain to `-∞`, and it is possible that *no* valid split exists for that level — a stricter outcome than the asymmetric per-node case. The fallback must be graceful: raise `max_bin` on the constrained feature (the standard mitigation; histogram binning can wipe out feasible thresholds) or allow the tree to **terminate early** at depth `<3` (a legitimate, fANOVA-friendly, lower-order outcome that the build code path must support as a tested case, not an error). (2) The hist/`max_bin` "shallow tree" caveat that bites ordinary GBMs is **largely moot here**: depth is fixed at 3 and a constrained feature occupies at most one of three shared levels, so split-starvation costs at most one level, never a tree collapse.

**fANOVA verdict: preserves exactness.** Constraints only restrict leaf *values*; the tree is still depth-3 oblivious, ≤3 features, constant leaves. The resulting 1D main-effect shape functions are monotone — exactly what actuaries want. **One subtlety to document:** monotonicity is guaranteed on the *summed score* (and on premium via the monotone log link) and survives purification on the *total function* and on the constrained 1D main-effect table — but it is **not individually guaranteed on each 2D/3D interaction table viewed in isolation** ([Lengerich purification](https://proceedings.mlr.press/v108/lengerich20a/lengerich20a.pdf) preserves the total function, [monotone ⟹ monotone first-order effects](https://arxiv.org/abs/2410.19098), but higher-order purified slices can be non-monotone alone). State this in table-export docs so exported tables aren't misread.

**Newton consistency and API.** Check the child-value ordering using the **2nd-order leaf weight `w = -G/(H+λ)`**, consistent with the per-level Newton gain pattern-boost uses elsewhere — not a first-order proxy. Expose constraints as a `name → sign` dict (not positional), because wrong-sign mis-specification is a large, silent accuracy hit (2.5–21.2% AUC reported in the literature; <0.2% is the *cost of correct* monotonicity on large pricing data, [Henckaerts et al.](https://arxiv.org/abs/1904.10890)).

**The `intermediate`/`advanced` refinement is mostly degenerate at depth 3.** Bartley et al.'s [tighter bound-relaxation methods](https://arxiv.org/pdf/2011.00986) (LightGBM `monotone_constraints_method`) recover accuracy that the over-constraining `basic` method throws away. But they exploit *deep, asymmetric, leaf-wise paths where a feature is split at many nodes* — they memorize subtree min/max and full path `(feature, threshold)` combinations. In a depth-3 oblivious tree each constrained feature is split **at most once** (one shared threshold), so the path combinatorics the refinement relies on are largely degenerate and `basic` over-constraining is far milder. Ship `basic`-style enforcement in v1; treat intermediate/advanced as a v1.5/research item gated on benchmarks actually showing a measurable monotonicity tax at depth 3. Note XGBoost ships only `basic`, another signal the refinement is a minor LightGBM-specific gain. `monotone_penalty` (forbid constrained splits in the first `floor(X)` levels) is even blunter here: with only levels 0/1/2, `X≥1` forbids the root split, `X≥3` disables the constraint entirely (validate/clamp `X<3`), so the useful range collapses to roughly `X∈(0,2)` — research-priority polish at best.

### Interaction constraints — the headline structure lever, with *stricter* semantics than XGBoost

This is the technique most aligned with pattern-boost's thesis: by restricting which features may co-occur in a tree, you decide **exactly which pairwise/triple tables can be non-zero**, directly sculpting the fANOVA structure while preserving exactness and order-≤3. CatBoost notably *lacks* this — it relies on auto-combinations + post-hoc importance — so pattern-boost should **borrow from XGBoost/LightGBM, not CatBoost** ([XGBoost interaction constraints](https://xgboost.readthedocs.io/en/stable/tutorials/feature_interaction_constraint.html)). Two disjoint groups ⇒ no `f_ij`/`f_ijk` mixing those features can ever be non-zero; this is the desired guarantee and should be a **tested invariant of the table export**.

**The critical design call: prefer stricter whole-tree semantics over XGBoost's union rule.** XGBoost enforces interactions *per path*: when a feature splits a node, descendants may use the **union** of all groups containing the ancestors. This UNION semantics lets a feature appearing in multiple groups have descendants spanning a union not contained in any single group — so one XGBoost tree can end up using a feature triple that is *not jointly within any permitted group* (overlapping groups like `[[1,2],[2,3,4]]` silently allow `{1,3,4}` via the bridge feature `2`). For pattern-boost, where each tree's ≤3 features become *one exact 3D table an actuary reads as a single rating cell*, this is wrong: you want "permitted interactions" to exactly match "tables that can appear." Adopt the **stricter rule**: require the tree's realized feature set to lie *jointly within one allowed group*, enforced by intersecting the live candidate-feature set at each of the 3 levels given the features already chosen. Because all paths in an oblivious tree share the same 3 splits, this is *cleaner* than XGBoost's per-path logic — you check the constraint once per tree's feature set, not per path.

**Mechanism in the oblivious build.** At level `L`, restrict candidate split features to those that keep the running feature set inside some single allowed group (incremental candidate **masking**, not post-hoc rejection). The constraint is per-tree, so across *different* trees a feature may belong to multiple groups and the ensemble can still realize any interaction permitted by *some* group — only co-occurrence *within a tree* is constrained.

**Implementation pitfalls.** (1) **Level starvation:** candidate restriction can leave no admissible `(feature, threshold)` that keeps the set inside any group, especially at level 3 — handle gracefully by terminating the tree early (a valid lower-order tree), never erroring or forcing a bad split. (2) Compose with monotone constraints and missing-value default-direction handling and verify *jointly* — both operate on the same per-level candidate set. (3) Normalize/validate user groups and surface the *realized* allowed feature-set lattice, since overlapping groups can silently merge.

**`max_interaction_order` and the GAM/GA2M mode.** Expose a `max_interaction_order` knob (1 = additive GAM / only 1D tables, 2 = GA2M-style, 3 = full) layered on the explicit whitelist — this is exactly the lever the [GAMI-by-adapting-XGBoost literature](https://arxiv.org/abs/2309.02426) uses to build structured additive models, which is pattern-boost's interpretability target. **Singleton-group additive mode** (each feature in its own group, `max_interaction_order=1`) forces a fully additive GAM: every tree is restricted to a single feature, yielding pure 1D main-effect step-function tables — the regulator-facing "no interactions" rating mode, and a headline feature. Implement via incremental candidate masking, not post-hoc rejection; with singletons all 3 splits land on one feature (still oblivious, a 1-feature step function), so benchmark the accuracy loss from forced additivity. This is `preserves_exactness` — it *restricts* which tables form, sharpening the decomposition.

**Operations order.** When penalties and constraints both apply: **reject constraint-violating candidates FIRST, then apply feature weights/penalties to the survivors** — so a penalty can never resurrect a constrained-out split.

### Feature penalties & cost-aware splitting — parsimony as a score modifier

CatBoost's penalty family (`feature_weights`, `first_feature_use_penalties`, `per_object_feature_penalties`, scaled by `penalties_coefficient`) and LightGBM's CEGB (`cegb_tradeoff`, `cegb_penalty_feature_coupled/lazy/split`) all operate by **adjusting the per-level split score** — directly compatible with the oblivious summed-gain search, no table-form change, `preserves_exactness` ([CatBoost params](https://catboost.ai/docs/en/references/training-parameters/common), [CEGB advanced topics](https://lightgbm.readthedocs.io/en/latest/Advanced-Topics.html)).

- `feature_weights`: multiply candidate score by a per-feature weight; `0` ⇒ feature drop. Apply the multiplier to the **gain magnitude (positive)** so down-weighting behaves monotonically — multiplying a signed/near-zero gain can invert ordering near zero (unit-test this against the table-reconstruction invariant).
- `first_feature_use_penalties`: subtract a one-time penalty the first time a feature enters the model. This biases toward parsimony — **it does not hard-cap feature count**: with hundreds of trees a useful feature usually pays back its one-time penalty and still enters. For a strict cap you need an explicit max-features constraint; this is a *soft prior*.
- `per_object_feature_penalties` / CEGB lazy cost: model real per-quote feature-acquisition cost (e.g. external bureau/credit data charged per quote), steering the model to reuse already-acquired cheap features. Genuinely useful for insurance, but niche → research.

**The oblivious caveat on CEGB.** A feature chosen at level 1 is shared by all 8 leaves, so "first use of a feature" and "`num_data_in_leaf`" mean something different than in LightGBM's leaf-wise growth: the coupled penalty collapses to a flat per-feature tax on the global per-level search. That is fine and exactness-safe, but it's no longer "CEGB" in any meaningful sense — implement it natively as a per-feature score tax, not by porting the upstream code (which also carries a known crash, [LightGBM #5317](https://lightgbm.readthedocs.io/en/latest/Advanced-Topics.html)).

These are **global, user-supplied domain/cost priors**, not auto-tuned; document that they trade a little accuracy for parsimony/cost-awareness and that the resulting model stays exactly decomposable — the tables are simply *sparser*. Fewer active features ⇒ fewer, cleaner tables for an actuary, which is high-value for pattern-boost → v1.5.

### Interaction-order control & sparsity: FAST screening + heredity

Beyond hard interaction whitelists, two train-time selection techniques keep the *realized* triple-table set small and meaningful — central to "a rating table you can read."

**FAST interaction screening** ([Lou et al., KDD 2013](https://www.cs.cornell.edu/~yinlou/papers/lou-kdd13.pdf)) ranks candidate feature pairs by the RSS reduction of the simplest 4-quadrant (one cut per feature) predictor fit on the residual, using cumulative target/weight histograms so each pair is `O(1)` after an `O(b²+N)` setup. The numerator `[PD_jk − PD_j − PD_k]` *is* the centered pairwise component, which dovetails with pattern-boost's binned/histogram engine. Use it as a **pre-filter feeding interaction constraints**: admit a triple only if its sub-pairs cleared a FAST threshold.

Three honest caveats: (1) **FAST is intrinsically pairwise** — a literal triple-FAST is `O(b³)` per triple × `C(n,3)` triples and is research-only; the sound cheap path to triples is **heredity/composition** (admit a triple only if its constituent pairs are admitted). (2) FAST ranks on *RSS-on-residual*, a different objective than the booster's Newton summed gain — use it as a **soft candidate-pruner**, never a hard gate, with final selection by **exact purified Sobol variance after training**. (3) Don't oversell speed: the per-level oblivious search is already cheap; the value is interpretability/table-parsimony + mild regularization, and it must run on residuals *after* main effects (which biases toward fewer interactions — benchmark on correlated pricing data before claiming an accuracy win).

**Heredity / strong-hierarchy admission** ([GAMI-Net](https://arxiv.org/pdf/2003.07132), [GAMI-Tree](https://arxiv.org/pdf/2207.06950)): admit an interaction only if (weak) at least one, or (strong) both/all, of its lower-order parents are already important. Maps onto pattern-boost as an interaction-constraint rule on which feature triples a tree may use, keeping the triple-table set both small and hierarchically coherent (no "orphan" high-order effects with no main-effect support — usually spurious). It's `adaptable` (not free post-hoc) because interactions emerge from co-splits inside the stateful split-finder, so heredity must gate the candidate set there. Train-time only ⇒ exactness untouched. The 3rd-order extension is a reasonable but **unvalidated** extrapolation of the pairwise GAMI literature — it needs its own benchmark, and the "importance" threshold + admission ordering interact with the boosting schedule (too strict silently caps achievable accuracy).

**A design warning from GAMI-Tree** ([Hu et al.](https://arxiv.org/abs/2309.02426)): the EBM-style **two-stage** recipe (fit all mains to convergence, *then* fit interactions on the residual) can fail to converge / mis-attribute mass when mains and interactions are correlated. Prefer **single joint boosting over the realized ≤3-feature supports followed by one purification pass** rather than a naive "freeze mains → boost pairs → boost triples" pipeline.

### Forced splits and forced bins — mandating the table grid

Two distinct LightGBM features ([Parameters](https://lightgbm.readthedocs.io/en/latest/Parameters.html)), with very different risk profiles.

**Forced bins (`forcedbins_filename`)** pin histogram boundaries to chosen values (regulatory age bands, round numbers). This maps cleanly onto the binning the engine already needs, lets pricing tables snap to mandated band edges, is accuracy-neutral, and is fully `preserves_exactness`. **Ship this as the primary, safe usability win** — make the pattern-boost equivalent take `(feature, border-value)` and feed it into the global shared-border set. This is the load-bearing half: to truly *pin* a boundary you need forced bins, because a forced split threshold otherwise snaps to the nearest existing `bin_upper_bound ≥` the request ([issue #1829](https://github.com/microsoft/LightGBM/issues/1829)).

**Forced splits (`forcedsplits_filename`)** are trickier. The LightGBM JSON schema is a **per-node, arbitrarily-nested asymmetric tree spec** — structurally incompatible with oblivious depth-3 trees; do **not** port the schema. Only a re-specified, pattern-boost-native "force the *shared* split at level `L` to `(feature, border)`" is adaptable. Two corrections to the common description: LightGBM does **not** apply forced splits unconditionally — it *skips* them when the gain worsens ([issue #1783](https://github.com/microsoft/LightGBM/issues/1783), warning "Gain with forced split worse than without split"); and the upstream per-node forced-tree path has known stability bugs when combined with feature subsampling or high-cardinality categoricals ([#4286](https://github.com/microsoft/LightGBM/issues/4286)/#4591/#4601). For pattern-boost, make any forced shared split **gain-aware** (warn/surface when a forced split has negligible gain so it doesn't silently waste one of only 3 levels) and validate against interaction-constraint groups (forcing a feature into a low level can conflict with its allowed group). Priority v2; this is the lower-value, higher-risk half of the bundle.

### Smoothness: the one place that *breaks* exactness — keep it out of the lossless model

Actuaries want smooth relativities "for commercial or statistical reasons," but smoothness is fundamentally at odds with piecewise-constant lookup tables. Three routes appear in the list; understand precisely what each costs.

**`linear_tree` / `linear_lambda` (piecewise-linear leaves)** ([LightGBM](https://lightgbm.readthedocs.io/en/latest/Parameters.html), [GBDT-PL / Shi et al.](https://arxiv.org/abs/1802.05640)). If leaf regressors are constrained to ⊆ the tree's own ≤3 split features (the GBDT-PL "incremental feature selection" variant), the ensemble stays a sum of ≤3-feature functions, so **order-≤3 truncation survives** — but each table cell becomes a piecewise-**linear** shape function (`x ↦ a + bx`) instead of a constant relativity. That is a deliberate, honest **`changes_table_form`**, not a break of exactness. Two hard blockers for pricing: (1) LightGBM monotone constraints bind only the *splits*, **not** the leaf linear coefficients, so a "monotone" model can be non-monotone *within* a cell — unacceptable for regulated rating ([the constraint gap is documented upstream](https://lightgbm.readthedocs.io/en/latest/Parameters.html)); and (2) the **production LightGBM `linear_tree` uses ALL leaf-branch numerical features**, which couples >3 features per tree and would *break* order-3 and the additive-table decomposition (empirically: contributions don't sum, [#4002]). Only the ≤3-feature-restricted GBDT-PL variant is even a candidate, and it would require re-deriving Lengerich purification for affine-per-cell tensors plus slope clipping to bound tail relativities. Verdict: **skip / research**, gated behind an explicit "shape-function table" mode, never default.

**Post-hoc Whittaker–Henderson table graduation** ([WH graduation](https://www.casact.org/sites/default/files/2021-01/05-Goldburd-Khare-Tevet.pdf), used as the smoother inside [ICEnet](https://arxiv.org/abs/2305.08807)). Because pattern-boost's tables *are* the exact shape functions, you can smooth each extracted table by minimizing `Σ_k w_k(v_k − s_k)² + h·Σ_k(Δ^z s_k)²` (closed-form banded solve; `z`=2 or 3), with optional monotone constraints via a constrained QP. This is clean post-processing — no NN, no retraining, no split-objective change — and preserves order-≤3. **But it changes table *values* (lossy), so the smoothed tables no longer sum bit-for-bit to the raw ensemble: this is `breaks_exactness` of the lossless-equivalence invariant.** Ship it as an explicit, separately-named **"deploy smoothed tables" mode** distinct from the audited lossless model, never folded silently into the export. Do **not** relabel it `changes_table_form` to keep it in a lower tier — the distinction is precisely that values change while form (constant cells) does not, and the lossless reconstruction test must fail by design for the smoothed artifact.

**ICEnet in-training smoothness/monotonicity penalties** ([Richman & Wüthrich](https://arxiv.org/abs/2305.08807)). The pseudo-data ICE-curve penalty (squared third differences of predicted ICE values; ReLU hinge on first differences for monotonicity) is **gradient-based and natural only for NNs** — it cannot enter the closed-form Newton per-level split objective of an oblivious GBM without destroying it. **Incompatible / skip.** The only pattern-boost-shaped salvage is the post-hoc WH route above (or a *different*, research-tier idea: a soft squared-third-difference penalty on adjacent table cells applied as a post-fit graduation — same lossy caveat). The hard-monotonicity half is already fully covered by split-rejection + bound-propagation.

**Note on the cheap, exact alternative:** for smoother tables *without* leaving the lossless model, the in-spec levers are **finer binning / more trees** and `path_smooth`/`L2` shrinkage of leaf values — none of which change table form.

### Leaf-value regularizers that shape table content (not structure)

These act only on the 8 leaf values, so structure, ≤3-feature property, and exactness are all untouched — but they materially affect *rating-table quality*.

**`path_smooth`** ([LightGBM](https://lightgbm.readthedocs.io/en/latest/Parameters.html)) shrinks each leaf toward its parent, weighted by sample count: `w_node = w·(n/path_smooth)/(n/path_smooth+1) + w_parent/(n/path_smooth+1)`. For depth-3 oblivious trees this regularizes the 8 leaves toward their depth-2/1 ancestors — directly stabilizing thin/low-exposure cells (rare risk factors), the exact pain point in insurance rating tables. It is "adaptable" (not free `native`): you must construct the recursive root-to-leaf parent path in the oblivious builder. Two cautions: the prior is the *ancestor split-node value*, **not** a designed actuarial credibility complement — treat `path_smooth` as a *foundation* for a credibility-shrinkage feature, not its final form; and verify that shrinkage toward the parent cannot nudge a leaf across a monotonicity bound (enforce constraints *after* smoothing). Implement from the code form, not the old docs ([#5139 discrepancy](https://github.com/microsoft/LightGBM/issues/5139)). v1.5.

**L2 leaf reg (`lambda`/`l2_leaf_reg`)** is core v1 — it's already in the Newton leaf math (`w = -G/(H+λ)`, gain `Σ G²/(H+λ)`). It *does* affect which split is selected (it sits in the gain denominator), so it is not pure post-hoc shrinkage, but the chosen split is still a depth-3 oblivious ≤3-feature constant-leaf tree ⇒ exactness intact. Smaller/smoother relativities also make exported tables more stable. Re-validate the good default (~1–3) on Tweedie/Poisson losses, since the effective strength shifts under the log link and exposure offsets.

**L1 leaf reg (`alpha`, soft-thresholded leaves)**: `w* = -T(G,α)/(H+λ)`, `T(G,α)=sign(G)·max(|G|-α,0)`, zeroing weak leaves. **Critical caveat for pattern-boost:** the "sparser tables / base-relativity 1.0" reading is **invalid** here — purification redistributes raw-leaf mass, so L1 sparsity at the *raw-leaf* level does **not** propagate to the *purified deployed* tables. Do not market L1 as a table-simplification lever; real table sparsity comes from FAST/heredity/purified-variance pruning. v2, parameter-parity at best.

**`max_delta_step`** (the Poisson safeguard) is v1: it caps the Newton leaf step so the exp-link's exponential gradient can't overshoot. Two distinct mechanisms hide under the name — XGBoost's **hard clamp** `|w| ≤ max_delta_step` (objective-agnostic, post-hoc, changes only leaf values) and LightGBM's **Poisson hessian inflation** `h = μ·exp(δ)` (default `0.7`, smooth, objective-coupled, *changes which trees are built* because it changes split gains). They are **not interchangeable**. For pattern-boost's log-link pricing objectives, make the **hessian-inflation route the default for `count:poisson`** (LightGBM-parity, Newton-consistent, composes cleanly with monotonicity bound-propagation), and optionally expose the generic clamp (useful for imbalanced logistic, recommended 1–10). If both are enabled, define ordering — **enforce monotone bounds *after* clamping** — and test that monotonicity still holds as an invariant, since a hard-clamped output can land off the propagated weight bounds. Keep the global default `0` (off) to preserve squared-error/AUC benchmark parity; auto-set `0.7` only for Poisson/count, as both incumbents do. Either way the lossless table export is unaffected.

**Split-quality stops** (`min_data_in_leaf`, `min_sum_hessian_in_leaf`, `min_gain_to_split`) shape *which* cells exist and at what credibility. Two pattern-boost-specific notes: the floor must hold across **all 8 cells of a shared level** (a single under-supported cell vetoes the level — stricter than asymmetric trees, and exactly the credibility guarantee actuaries want), and pattern-boost should enforce `min_data_in_leaf` **exactly** on the binned count the histogram already carries (LightGBM's is hessian-approximate and occasionally violated — exactness is free here and more defensible to a regulator). Under Poisson with an exposure offset, `min_sum_hessian` floors `Σ h = Σ e·μ` (effective-sample-size / Fisher-information mass), which **drifts every iteration** and is *not* a literal exposure floor — for a stable credibility floor an actuary expects, prefer `min_data_in_leaf` on raw count or add a **weight-sum (`Σ e`) floor**. `min_gain_to_split` lets a level fail and the tree terminate at `<3` features — a legitimate, fANOVA-friendly lower-order outcome (a no-op constant tree just contributes to `f_0`); default it to `0` for incumbent parity, and apply the threshold *after* any constraint-driven `-∞`.
