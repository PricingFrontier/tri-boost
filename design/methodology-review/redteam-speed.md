# Red-Team Critique — Cluster: SPEED REALITY

**Verdict in one line:** The inference story is genuinely strong and mostly holds. The *training-speed* story is honestly framed ("parity, not a win") but is built on a **load-bearing design choice that is conflated with a product requirement** — and that choice is the single biggest unforced speed loss in the spec. Two hidden inference costs are under-counted.

---

## 1. The depth-as-3-splits / oblivious choice — the big one (SHOULD CHANGE, as a benchmarked fork)

**The conflation, confirmed in the contract.** Skeleton §3/I1 *defines* the invariant as `depth ∈ 1..=3` **and** `splits.len() == distinct_raw_features`. That second clause is not required by Aim 2. Exact ≤3rd-order decomposability requires only **≤3 distinct raw features per tree**. It does *not* require exactly-3 splits, nor symmetric structure, nor one split per feature. A tree on ≤3 distinct features with a feature *repeating* across levels (e.g. splits on `age`, then `age` again at a finer border, then `region`) is still a piecewise-constant function of {age, region} — its fANOVA truncates at order 2, exactly. Likewise a depth-4 or depth-5 oblivious tree constrained to ≤3 *distinct* features is still ≤3rd-order. The spec has quietly welded a **capacity choice** (8 cells, 3 splits) onto the **fixed cap** (3 features).

**Why this is a speed problem, not just an accuracy problem.** The whole training-speed risk is self-identified in the brainstorm: *"Depth-3 trees too weak → needing so many trees that speed parity fails."* The spec's hedges (quantized histograms ~2×, AGBM, multi-Newton, fully-corrective refit) are all attempts to *buy back* tree count that the 8-cell-per-tree cap throws away. But the cheapest tree-count reduction is sitting in plain sight: **let each tree spend more splits on its ≤3 features.** A tree with, say, up to 6 splits over ≤3 distinct features has up to 64 cells instead of 8 — far finer per-feature resolution and far more interaction cells, i.e. a much stronger per-tree learner — at *no cost to decomposability* (still ≤3 features ⇒ still ≤3rd order). Stronger trees ⇒ fewer trees ⇒ less wall-clock for equal accuracy ⇒ the "parity" claim gets easier, not harder. This directly attacks the one named training-speed risk.

**Aim impact.** Predictiveness: up — a depth-3-feature/6-split oblivious tree is strictly more expressive than a 3-split one and recovers part of the per-tree-weakness gap to CatBoost's depth-6 (research/01 explicitly flags depth-3 as *more aggressive* than CatBoost's depth-6 default). Speed (training): up via lower tree count — though each tree costs more (more levels = more histogram passes; histogram tensor grows to 2^d leaves). Speed (inference): roughly neutral — more cells per table but the LUT-sum is unchanged in *form*; tree-walk path A grows linearly in depth (6 compares vs 3). **Decomposability: fully preserved** — this is the key point; the invariant just needs re-specifying as "≤3 distinct raw features," dropping the "== splits.len()" clause.

**Cost / honest caveat.** This is not free. (a) More levels = more histogram-build passes per tree, and the subtraction trick saves less when leaf counts grow; the *per-tree* cost rises, so the net training win depends entirely on the tree-count drop exceeding the per-tree cost rise — **must be benchmarked**, not assumed. (b) It enlarges the merged grid (see §3 below) and the 3-D tensors, raising purification and table memory. (c) Monotone bound-propagation at depth >3 stops being degenerate (§07.5 currently leans on "each constrained feature splits at most once"). 

**Recommendation:** Do not lock 3-splits-oblivious as *the* structure. Re-state I1 as the ≤3-distinct-feature property only, and make **max splits per tree** (and symmetric-vs-mildly-asymmetric) a benchmarked config knob, default to the current 3 for v1 but with the depth-6/≤3-feature variant as the first thing tested in §14. The spec already gates AGBM and fully-corrective-refit on benchmarks for exactly this "shrink tree count" goal; this lever is cheaper to build than either and attacks the same target more directly. **Steelman of the current choice:** 8-cell trees give the smallest possible tables, the simplest monotone clamp, trivially cache-resident histograms, and a clean `[f32;8]` LUT — real engineering virtues. But "smallest tables" is an Aim-2-*adjacent* nicety, not Aim 2 itself, and it is being paid for in tree count, i.e. in Aim 3. The spec should stop treating it as forced.

## 2. Quantized-integer histogram + oblivious — is it the fastest path? (HOLDS, with one caveat)

**This holds up and is well-argued.** Quantized i64 integer histograms doing double duty as the bit-reproducibility mechanism *and* the ~2× speed lever (§06.3, §11.1) is the right call — it is exactly LightGBM 4.x's quantized-training result (research/02 §2). The associativity-gives-reproducibility argument is correct and elegant; it makes the determinism [GATE] essentially free rather than a tax. Per-thread padded private histograms + fixed-order reduce (§11.3) is the standard high-performance pattern with no Mutex. Column-major u8, feature-parallel search, subtraction trick: all correct, all matching production GBMs.

**Is there a *structurally* faster training design?** Within the histogram-GBM family, no — this is the frontier. The honest answer the spec should internalize: the structurally faster paths are the ones it has (correctly) ruled out of scope — **GPU training** (CatBoost reports ~40× CPU→GPU; the oblivious structure is unusually GPU-friendly, research/01 §2) and **GOSS-style aggressive row subsampling**. MVS is chosen over GOSS (defensible — dominates on split-score variance), but MVS at `subsample=1.0` default (§06.1) samples *all* rows, so the row-count win is opt-in and off by default. **Caveat / recommendation:** the spec under-uses its own cheapest speed lever — set a **subsample default < 1.0** (e.g. 0.5–0.8) so MVS actually cuts histogram-build rows out of the box; this is pure speed with a mild regularization upside, and every incumbent ships row subsampling on by default in their fast configs. The `Backend` seam for GPU-later (§02) is the right place to bank the real structural win when it comes.

## 3. Inference: LUT-sum / branch-free claims — REAL, but two hidden costs

**The branch-free 8-cell claim is real and excellent (HOLDS).** Path A (§10.2) — 3 compares → 3-bit index → one `[f32;8]` read, branch-free, SIMD-across-rows — is precisely CatBoost's documented inference advantage (research/01 §2: ~35× XGBoost, ~83× LightGBM single-thread, with the honest caveat that multipliers shrink against multi-threaded competitors). This is the spec's bankable structural win and it is not overstated. Path B (table-sum, cost independent of tree count) is a genuine second win for dense ensembles. Both are correctly load-bearing-tested for equality.

**Hidden cost 1: merged-grid blow-up is under-counted (real risk, needs a guard).** §08.1/§08.9 assert merged-grid cells stay "a few hundred per axis at most." This is the **accumulation of distinct realized thresholds across all trees** — with `max_bin=254`, 1000–4000 trees, and an axis that is split at many different borders across the ensemble, the union grid for a *hot* feature can approach its full bin cardinality (up to ~254 cells), and a 3-D table on three such axes is `254³ ≈ 16M f64 cells = 130 MB` for a *single* triple. The spec's "few hundred per axis" is plausible for cold features but optimistic for the 2–3 features the booster splits on most (which are precisely the ones that form the high-Sobol tables you most want). **Compounding factors:** my §1 recommendation (more splits/tree) makes this *worse*; the ensemble path (§09.5) takes the **union across K members**, worsening it again; Nesterov/refit shrink tree count but not necessarily distinct-border count. This is a Path-B inference cost (bigger tensors to read, worse cache) and a memory/purification cost, not a Path-A cost. **Fix:** the spec needs an explicit **merged-grid coarsening / threshold-quota** policy (cap distinct borders per axis in the bank, snapping near-coincident realized borders) with an exactness-preserving error bound, OR a documented hard ceiling that flips to a sparse tensor representation above a cell threshold. Right now the blow-up is asserted away rather than bounded.

**Hidden cost 2: purification time and the f64 fixpoint.** Purification is single-pass and cheap under the default `ProductMarginals` `w` (§08.3) — correct and well-chosen. But (a) under `Joint` `w` it iterates to a fixpoint (up to `max_iter=64`), and (b) purification cost is `O(#cells)`, so it inherits the §3 blow-up directly. For the **ensemble** path the bank is purified K times plus once for the union (§09.5), and `OuterBag{8}` purifies 8 banks. This is "cheap relative to training" *per the spec* — true for a single small bank, less true for an 8-bag union over hot features. It is a one-time post-fit cost, so it does not hit serving latency, but it does hit the `explain()` / export wall-clock and should be benchmarked, not assumed cheap. **Recommendation:** state a purification time budget in §14's yardstick and measure it on a deliberately border-rich fixture.

**Hidden cost 3: the teacher.** Distillation (§09.2) is correctly framed as caller-borne (the user supplies `teacher_raw`), so it costs the library nothing at train time and nothing at inference (teacher never enters the model). That framing is honest. The only thing to flag: the *default teacher is CatBoost*, which itself can train *slower* than LightGBM (research/01 §2). So the end-to-end "fit a distilled pattern-boost model" wall-clock, as a user experiences it, is **teacher-train + student-train**, and the spec's "competitive with LightGBM" claim is about the student only. This is defensible (distillation is opt-in, v1.5, default off) but the user-facing docs should not let "fast training" and "distillation mode" sit in the same sentence without the teacher-cost asterisk.

## 4. Front-end overheads (FAST / triple detector) — HOLDS

§07.8's complexity table is honest: heredity-by-composition avoids the `C(n,3)` scan, FAST is `O(b²)`/pair on histograms the engine already maintains, `constrain_candidate` is `O(1)`/candidate. The triple detector is *off the inner loop* (one dense-reference fit, pre-training) and default-off (§07.10). None of these touch the row-count-independence of the split search. The one cost is the optional triple-detector's dense-reference fit — but it is opt-in and benchmark-gated for exactly that reason. No change needed.

---

## Bottom line

| Decision | Verdict | Action |
|---|---|---|
| **3-splits-oblivious welded to the order-3 cap** | **Should change** | Re-spec I1 as "≤3 distinct raw features" only; make max-splits-per-tree a benchmarked knob. Highest-leverage training-speed lever in the spec; **decomposability-safe**. |
| Quantized i64 histogram + fixed-order reduce | Holds | Keep. Frontier-correct, dual-purpose. |
| `subsample=1.0` default | Should change | Default MVS subsample < 1.0 — free training speed, currently left on the table. |
| Branch-free 8-cell LUT inference | Holds | Keep — the real, un-overstated structural win. |
| "Few hundred merged cells/axis" assumption | At risk | Add an explicit merged-grid coarsening/quota policy with an exactness bound; the blow-up is asserted away, not bounded. |
| Purification "cheap" | Mostly holds | True on default `w`; benchmark it on border-rich + K-bag fixtures, budget it in §14. |
| Distillation teacher cost | Holds (honesty gap) | Keep; just asterisk "fast training" wherever distillation is mentioned. |

The spec's honesty ("parity not a win" on training, structural win on inference) is the right posture and mostly earns it. The one place it is **optimistic by construction** is the unexamined assumption that depth-3 *must* mean 3 splits — that conflation costs tree count, which is the exact quantity every other speed hedge in the spec is fighting to recover. Fix that conflation first; it is cheaper than AGBM, cheaper than fully-corrective refit, and it is decomposability-neutral.
